## Planning-Phase Drift Concerns

Monitoring is active for the following areas during planning and design:

### 1. Approval-Reuse Detection Correctness

**Concern**: Normalization rules for approval-evidence matching must follow the approved spec without silent semantic drift.

**Specification Authority**:
- `spec.md` § "What counts as duplicated approval evidence?" — "Quotes that match after whitespace normalization and markdown-emphasis stripping, unless the artifact explicitly declares a blanket multi-iteration authorization scope."
- `FR-003` — Detect reused approval evidence across iterations

**Drift Triggers**:
- Whitespace normalization includes or excludes cases not mentioned in the spec (e.g., line-break handling, tab expansion, multiple-space collapsing)
- Markdown-emphasis stripping incorrectly handles nested emphasis or non-emphasis markdown constructs (e.g., `**bold _nested italic_**`)
- Blanket-scope exemption criteria drift from "explicitly declares a blanket multi-iteration authorization scope"
- Quote matching allows approximate/fuzzy matches instead of exact post-normalization matching

**Evidence**: Fixtures and assertions in `tests/integration/validator-hardening-iteration2.ps1` will explicitly prove normalization behavior and blanket-scope handling before implementation lands.

### 2. Over-Claim Detection Correctness

**Concern**: Closeout-evidence validation and dirty-tree enforcement must follow the spec's exact file scope and evidence requirements.

**Specification Authority**:
- `spec.md` § "What file scope counts for the FR-004 working-tree-differences over-claim check?" — "Dirty-tree enforcement is limited to the iteration directory's canonical artifacts. Repo-level governance traces such as `.squad/decisions.md` and `.squad/identity/now.md` may still be consulted as evidence sources, but they do not fail the dirty-tree check by themselves."
- `FR-004` — Block unsupported closeout claims; enforce required review, retro, hardening evidence; limit dirty-tree scope to iteration directory
- User Stories § US4 acceptance scenarios — closed-status iterations must have accepted review, recorded retrospective, post-implementation hardening-gate evidence, and clean iteration-directory tree

**Drift Triggers**:
- Dirty-tree filtering includes repo-level changes (`.squad/decisions.md`, `.squad/identity/now.md`, top-level config files) as blockable dirt
- Required-evidence checks omit review, retrospective, or hardening-gate post-implementation verification
- Closed-status regex or pattern matching becomes loose and matches partial keywords (e.g., "closing" matching "closed")
- Iteration-directory boundary definition drifts from the exact feature-branch iteration folder

**Evidence**: Fixtures covering clean iterations, dirty iteration-directory cases, repo-level changes, and missing-evidence scenarios will be proven in `tests/integration/validator-hardening-iteration2.ps1`.

### 3. Bookkeeping-Classifier Correctness

**Concern**: The `.github/copilot-instructions.md` classifier must correctly distinguish bookkeeping-only changes from behavior-affecting changes without over-matching or under-matching.

**Specification Authority**:
- `spec.md` § "Which `.github/copilot-instructions.md` changes are bookkeeping only?" — "Timestamp, `## Active Technologies`, and `## Recent Changes` updates generated by `/speckit.plan`; changes outside those areas are behavior-affecting."
- `FR-006` — Improve validator failure reporting and restart guidance
- User Stories § US5 acceptance scenarios — classifier correctly identifies bookkeeping-only diffs and triggers restart guidance only for behavior changes

**Drift Triggers**:
- Timestamp regex over-matches or under-matches YAML/markdown ISO 8601 timestamp formats
- Section heading matching treats `## Active Technologies:` differently from `## Active Technologies` (colon sensitivity)
- Diff filtering logic includes or excludes other `.specrew`-driven sections (e.g., autogenerated sections outside the spec-listed three)
- Generated-vs-behavior heuristic drifts from author intent (e.g., manual edits within `## Recent Changes` are incorrectly marked as "bookkeeping")

**Evidence**: Fixtures with timestamp-only, section-header-only, mixed, and behavior-only diffs will be exercised in `tests/integration/validator-hardening-iteration2.ps1`.

### 4. Corpus Graduation Completeness

**Concern**: Known-traps rows for approval-reuse, over-claim, and (canonically) canonical-schema and canonical-concern must be marked as validator-enforced and graduated into the corpus without stale guidance.

**Specification Authority**:
- `FR-007` — Improve validator failure reporting and restart guidance
- Tasks T017, T021, T027 — Mark specific rows in `.specrew/quality/known-traps.md` with citations to requirements, tests, and implementation files

**Drift Triggers**:
- Corpus row citation references non-existent test files or implementation paths
- Row marking timestamp or author details drift from the recorded iteration closure date
- Stale guidance text remains after validator enforcement lands
- Canonical-schema and canonical-concern rows (from iteration 001) are not carried forward into the final iteration-002 corpus graduation pass

**Evidence**: Planning artifacts explicitly list the graduated rows inside iteration 002 itself: T017 (approval-reuse), T021 (over-claim), and T027 (canonical-schema plus canonical-concern).

### 5. Regression Preservation

**Concern**: Iteration 002 rules must not regress iteration 001 canonical-schema enforcement, canonical-concern enforcement, or structured FAIL reporting.

**Specification Authority**:
- `FR-010` § slice 2 — Preserve `validate-governance.ps1` / `shared-governance.ps1` / `specrew-start.ps1` command surfaces
- User Stories § US1, US2 acceptance from iteration 001 — canonical rules must remain stable and green across both iterations

**Drift Triggers**:
- Changes to `shared-governance.ps1` or `validate-governance.ps1` accidentally modify iteration 001 rule logic (canonical-schema detection, canonical-concern ordering)
- Iteration 002 feature flags or feature-ordinal checks inadvertently gate iteration 001 rules behind feature 013 checks only (breaking pre-feature-013 grandfathering)
- Replay harness updates to iteration-hardening-iteration1.ps1 are omitted or drift from the stored baseline
- Structured FAIL output format changes for iteration 002 rules but iteration 001 rules remain in old format

**Evidence**: Pre-implementation baseline and post-iteration-002-implementation validation will re-run `tests/integration/validator-hardening-iteration1.ps1`, `tests/integration/quality-profile-foundation.ps1`, and repo-wide `validate-governance.ps1 -ProjectPath .`.
