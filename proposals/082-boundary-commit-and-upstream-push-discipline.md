# Boundary Commit + Upstream Push Discipline

## Why

Specrew claims discipline it doesn't enforce.

A grep across the entire methodology surface (2026-05-21) reveals **zero explicit instructions** anywhere in Specrew that tell the Crew:

1. "Commit your work at every lifecycle boundary."
2. "Push your branch to origin after each commit so it's backed up."

The only mention of "committed" in any Crew-governing surface is one line in the coordinator governance prompt (`specrew-governance.md` line 98):

> "After each **committed** boundary handoff, synchronize `Commit Reference` away from `pending`..."

This **presumes** the commit happened. It doesn't **instruct** it.

### Empirical evidence the gap matters

Multiple incidents in a single session (2026-05-21) demonstrate the cost:

| Incident | Failure mode | Cost |
|---|---|---|
| F-029 implementation work (in progress at proposal-draft time) | Implementation + new tests + extended test + CHANGELOG entry all uncommitted; branch tip still at branch-creation commit | If session crashes or worktree corrupts: hours of work lost. No remote backup. |
| F-024 iteration-001 `tasks-progress.yml` | Generated by Squad weeks ago, still untracked | Audit trail Specrew is supposed to govern by isn't durable |
| F-028 iteration-001 `tasks-progress.yml` | Same as above | Same |
| `.squad/decisions/archive/*` | 10 untracked files accumulated across closeouts | Same |
| Crew "cancelling" hang (earlier today) | Force-quit lost all uncommitted work; no upstream backup to recover from | Required full state reconciliation |

### Connection to broader methodology issues

This gap is upstream of (and partly explains) several queued proposals:

- **Proposal 077 (Session Resume UX)**: A significant fraction of resume-UX complexity exists because work-in-progress is uncommitted between sessions. If commits happened at every boundary, "resume" would just be "checkout the last commit."
- **Proposal 055 (Always-In-Flow Discipline)**: Implicitly assumes flow ends with committed evidence. Doesn't explicitly require it.
- **Proposal 030 (Quality Hardening Bundle)**: Validator-side enforcement of boundary commits is one of the natural sub-components.
- **F-029 (Baseline Hygiene)**: Fixes `baseline_commit_hash` refresh at every boundary. But if no commit happens at the boundary, the baseline tracking has nothing to track against.

### User direction (2026-05-21)

> "Where in Specrew we provide the instruction to commit before stopping in each gate? Where do we tell it to push to the upstream (github) to know that the work is backed up?"

Answer (today): nowhere. This proposal closes the gap.

## What — Three Tiers

### Tier 1: Text-Only Discipline (~5 SP)

Add explicit commit-and-push instructions to the Crew-governing surfaces. No runtime enforcement; relies on Crew adherence + the existing governance prompt's authority.

| Surface | Addition |
|---|---|
| `coordinator/specrew-governance.md` | New rule (numbered alongside existing 14A, 1, 2, 3, etc.): "At every lifecycle boundary, before invoking `Invoke-SpecrewBoundaryStateSync` or emitting the boundary handoff, commit the iteration's work-in-progress with a clear boundary-scoped commit message. After the commit, push the feature branch to origin so the work is backed up." |
| `agents/implementer/charter.md` | New responsibility: "Commits implementation work at the boundary before handoff. Every implement→review transition includes at least one commit; multiple commits showing the implementation arc are preferred over a single mega-commit." |
| `agents/spec-steward/charter.md` | New oversight: "Verifies that boundary handoffs include a commit reference (non-`pending`) before accepting boundary advancement. Flags WIP-in-working-tree as a boundary-discipline violation." |
| `agents/reviewer/charter.md` | New review check: "Reviewer's pre-merge review confirms all in-scope work is committed. WIP files at PR-open time are a hard reject." |
| `agents/retro-facilitator/charter.md` | New retro prompt: "Retro evaluates whether commits happened at every boundary. WIP-at-boundary count is a retro signal." |
| `docs/user-guide.md` | New section: "Boundary Commit Discipline" explaining the methodology and what to expect at each boundary. |

**Implementation: pure documentation/prompt edits. No code changes. Mirror across `extensions/` + `.specify/extensions/` for the squad-templates.**

### Tier 2: Validator Warning Layer (~6 SP)

`validate-governance.ps1` gains a soft-warning rule that fires when boundary-sync evidence is observed but the working tree has uncommitted changes to in-scope files.

| Component | Detail |
|---|---|
| **Rule ID** | `boundary-wip-uncommitted` |
| **Category** | `boundary-discipline` |
| **Severity** | `warning` (soft; doesn't block) |
| **Detection** | When `Invoke-SpecrewBoundaryStateSync` is observed in the audit trail and `git status --porcelain` for the iteration's in-scope paths is non-empty at the time of validation |
| **Output** | Validator emits: `WARNING [boundary-wip-uncommitted]: Boundary <X> recorded but the following files have uncommitted changes: <list>. Commit before claiming readiness.` |
| **Idempotency** | Re-running validator after commit clears the warning |
| **Composability** | This rule lives at the same surface as F-028's `pre-review-commit-gate` (which is an `error`-severity zero-diff blocker); they're sibling rules in the same `review-evidence-integrity` / `boundary-discipline` family |

**Tier 2 includes a unit-test harness for the rule** and an integration test that creates a fake iteration with WIP files, calls boundary-sync, runs the validator, and verifies the warning surfaces.

### Tier 3: Hard Enforcement + Auto-Push (~10 SP)

Boundary-sync helper refuses to advance if WIP exists. Auto-push hook fires after every commit (configurable).

| Component | Detail |
|---|---|
| **`Invoke-SpecrewBoundaryStateSync` WIP gate** | Before mutating state, check `git status --porcelain` for the project root. If non-empty for tracked files in the iteration's scope: throw with a helpful diagnostic listing the uncommitted files and the suggested `git add` / `git commit` invocation. |
| **Configurable strictness** | `.specrew/iteration-config.yml` gains `boundary_discipline.commit_required: strict\|warn\|off` (default: `strict` per this proposal). `warn` falls back to Tier 2 behavior; `off` is opt-out for transitional projects. |
| **Auto-push** | After any commit detected (post-commit hook OR boundary-sync-triggered), execute `git push origin <current-branch>` with the following safety rules: |
| | • Never auto-push to `main` or `master` (only to feature branches). Squad's own branch model already enforces this; the safety rule is belt-and-suspenders. |
| | • Never use `--force` or `--force-with-lease`. Auto-push is fast-forward-only. |
| | • If push is rejected (e.g., remote ahead), emit a warning and stop — do not attempt to fetch/rebase automatically. |
| | • Auto-push respects `.specrew/iteration-config.yml`'s `boundary_discipline.auto_push: enabled\|disabled` (default: `enabled` in new projects; `disabled` for projects without an `origin` remote). |
| **Conditional skip when no remote** | If `git remote` returns empty (no remote configured), auto-push silently skips with a debug message. Local-only projects work normally. |
| **First-push upstream tracking** | If the feature branch doesn't yet have an upstream, the first push uses `-u origin <branch>` to set tracking. Subsequent pushes are plain `git push`. |

**Tier 3 includes integration tests for**: WIP-refused boundary advancement, the `warn`/`strict`/`off` config matrix, auto-push success, auto-push rejection (no force), no-remote graceful skip, first-push upstream tracking.

## Acceptance signals

- **AC1 (Tier 1)**: Coordinator governance prompt (`specrew-governance.md` + mirror) contains an explicit rule instructing commit-at-boundary AND push-after-commit, named and numbered alongside existing rules.
- **AC2 (Tier 1)**: All five baseline agent charters (Implementer, Planner, Reviewer, Spec Steward, Retro Facilitator) carry an explicit commit-or-oversight responsibility about boundary commits.
- **AC3 (Tier 1)**: `docs/user-guide.md` has a "Boundary Commit Discipline" section explaining the methodology to downstream Specrew users.
- **AC4 (Tier 2)**: `validate-governance.ps1` includes a `boundary-wip-uncommitted` rule at `warning` severity. The rule fires when a boundary-sync is observed in the audit trail but `git status --porcelain` for in-scope paths is non-empty.
- **AC5 (Tier 2)**: Validator output for the rule includes the list of uncommitted files and a remediation hint (`git add ... && git commit ...`).
- **AC6 (Tier 3)**: `Invoke-SpecrewBoundaryStateSync` refuses to advance if WIP is present (in `strict` mode). Failure mode: helpful diagnostic, no state corruption, exit non-zero.
- **AC7 (Tier 3)**: Auto-push hook fires after every commit detected by the boundary-sync flow. Safety rules verified: never pushes to main/master, never uses `--force`, fast-forward-only, skips silently when no remote.
- **AC8 (Tier 3)**: `iteration-config.yml` gains `boundary_discipline.commit_required` and `.auto_push` keys with documented values (`strict`/`warn`/`off` and `enabled`/`disabled` respectively).
- **AC9 (all tiers)**: Tests cover positive (commit happened, push succeeded) and negative (WIP detected, push rejected, no remote) scenarios.
- **AC10 (dogfooding)**: After Tier 1 ships, run `validate-governance.ps1` on Specrew's own repo and observe whether the new instructions are followed by the Crew during the next feature lifecycle.

## Out of scope

- **Commit message templates / conventions** — separate concern; commit messages already follow Crew host conventions (Copilot/Claude/etc.). 082 specifies WHEN to commit, not WHAT to write.
- **Force-push handling** — never automatically. Force-push remains a manual operation requiring human authorization.
- **Branch creation** — covered by Proposal 036 (Lifecycle Branch Reconciliation). 082 assumes branches exist.
- **Merge-vs-rebase discipline** — covered by existing `[[feedback-pr-at-feature-close-sdlc]]` (merge-commit only) and Proposal 036.
- **CI integration** — auto-push triggers CI naturally; 082 doesn't add new CI infrastructure.
- **Co-author tagging** — handled by host conventions, not Specrew methodology.
- **Conflict resolution on push reject** — 082 stops on rejection; rebase/merge is operator-driven, not auto.
