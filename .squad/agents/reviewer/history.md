## Core Context

Foundational patterns from May 8-10 review cycles:

- **Contract-gap traceability (May 8)**: Gaps must land in three places: functional requirement, traceability coverage, evidence artifact. Single-layer presence is insufficient.
- **Artifact-truth in multi-iteration scopes (May 8)**: state.md stays iteration-local; deferred work appears on next iteration, not in "Tasks Remaining" of prior iteration.
- **Pre-review validation (May 8)**: Passing tests is insufficient; re-verify plan.md task statuses, phase metadata, and reviewer-packet scaffold before formal review.
- **Quality-gate closure (May 8)**: Non-mechanical gates must have Category metadata; Gap Ledger must be explicit (fixed/deferred) or blank ("No known gaps").
- **Planning-repair verification (May 8)**: Three-part check: feature plan expresses multi-iteration split, iteration artifacts exist and sync, and `validate-governance.ps1 -IterationPath` passes.
- **Fixture-setup truth (May 8)**: Prefer empty directories + lifecycle truth over speculative placeholders.
- **Phase 2 governance helpers (May 8)**: Centralize Markdown parsing + approval resolution; prove via integration path mixing deferrals and blockers.
- **Hardening-fixture approval (May 8)**: Pair hardening-gate.md with canonical decisions.md entry; otherwise it's a claimed defer, not an approved one.
- **Governance regression suites (May 8)**: Fork from passing baseline; vary only new artifact + approval evidence to isolate signal.
- **Hardening-gate contract states (May 8)**: Prove blocked/deferred-with-approval/ready separately via shared parser/approval helpers.
- **Validator-hardening alignment (May 8)**: Bind enforcement to Phase 2 metadata + shared hardening-state helper; align verdict names, paths, and approval semantics.
- **Fail-closed-gate dogfood (May 8)**: New fail-closed gates must pass under accepted review before retro; self-approved defer erases dogfood signal.
- **Infrastructure-slice hardening (May 8)**: Review only the bounded scope delivered (not deferred story work); use canonical v1 schema (five concerns).
- **Handoff-visibility test tracing (May 10)**: Integration tests must trace full scaffold/replay invocation, not just fixture files.
- **Multi-file task completeness (May 10)**: Task naming three files is complete only when all three are updated; verify each file's diff.
- **Foundational vs. story layering (May 10)**: Distinguish generic signal (foundational) from cap-specific fields (story); verify independently.

## Learnings
- 2026-05-15: Refreshing Feature 018's pre-implementation review after the iteration-scoped hardening gate landed confirmed a durable reviewer rule: once the authorized boundary artifacts exist under `iterations/<nnn>/`, the verdict should follow that iteration-scoped package and approval ledger, not stale feature-root planning references. The missing-gate blocker can legitimately collapse to `ready-with-concerns` when the gate exists, the decision ledger records sign-off/authorization, and governance validation passes; the remaining concern set must then travel forward as mandatory verification targets rather than rephrased as optional polish.
- 2026-05-15: Feature 018 pre-implementation review reinforced a durable reviewer boundary for renderer-heavy follow-up work: design readiness and implementation-boundary readiness are separate truths. A spec/plan/tasks package can be substantively implementable while implementation is still blocked because the hardening-gate artifact named by the plan and approval ledger does not yet exist. For rich-terminal features, the hardening gate must also record terminal-capability precedence explicitly so `--ASCII`/Unicode opt-outs, non-interactive output, UTF-8 checks, Windows VT fallback, and Unicode-preserving snapshot rules are reviewed as one chain instead of guessed during implementation.
- 2026-05-11: Approval-recording boundary audit revealed four recurring defect patterns: (1) state.md status lags behind hardening-gate signed verdict, (2) plan.md conflates planning-level approval with hardening-gate-triggered implementation authorization instead of recording both separately, (3) hardening-gate.md Approval Ref field left blank, violating traceability mandates, (4) status summaries undercount feature-specific concerns in the Concern Review table. These gaps point to a need for stricter artifact-truth validation before sign-off: hardening-gate.md Overall Verdict change requires cascading status updates to state.md and plan.md, and Approval Ref must be populated before gate sign-off is recorded as complete.

- 2026-05-11: Retrospective audits must verify that friction events (rejected boundaries, repair cycles, rework authorization) are named explicitly in the "What Didn't Go Well" or equivalent friction section, even when friction is subsequently resolved. Retrospectives that move repair narratives into "What Went Well" while claiming zero friction in the negative section are artifact smoothing and fail truthfulness audit. A rejected approval boundary plus an independent repair cycle in the same iteration constitutes friction that must be named; the retrospective may explain how it was resolved, but omitting the friction event itself from the negative section is incomplete truth.
