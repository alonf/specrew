## Learnings

- 2026-05-08: Contract-gap closure reviews should verify that each accepted gap lands in three places when relevant: a binding functional requirement, explicit traceability coverage, and a reviewable evidence artifact definition. Presence in only one of those layers is not enough to preserve enforceability.
- 2026-05-08: When an iteration scope is split across iterations, `state.md` must stay iteration-local. Deferred feature work belongs on the next iteration artifact, not in the prior iteration's `Tasks Remaining`, or lifecycle status and closure evidence drift immediately.
- 2026-05-08: For formal review/demo, passing slice tests is insufficient. Re-check the live iteration `plan.md` task statuses, phase metadata, and reviewer-packet scaffold path, because review readiness can still fail on artifact truth or helper crashes after implementation looks complete.
- 2026-05-08: Review closeout on Phase 1 quality slices is still blocked if the reviewer packet leaves non-mechanical gates `planned`. Add `Category` metadata to `Required Quality Gates`, rerun `scaffold-reviewer-artifacts.ps1`, and keep the Gap Ledger at `No known gaps remain.` or an explicit fixed-now/deferred classification.
- 2026-05-08: When a planning repair claims a previously missing iteration boundary is fixed, verify three things separately before clearing implementation: the feature-level plan/tasks now express a capacity-true multi-iteration split, the target iteration artifacts exist and stay synchronized, and `validate-governance.ps1 -IterationPath <iteration>` passes on that specific slice. If all three hold, the remaining blocker is coordination-only human execution approval, not another planning defect.
- 2026-05-08: When a fixture-only setup task is intentionally bounded before scenario content exists, prefer empty root directories plus precise lifecycle truth over speculative scenario placeholders; that keeps later task ownership and evidence expectations honest.
- 2026-05-08: For Phase 2 governance helpers, centralize Markdown-table parsing and approval-reference resolution together, then prove them with one integration path that mixes human-approved deferrals and still-blocking `tbd` concerns. That keeps later validator work focused on policy enforcement instead of re-solving artifact parsing.
- 2026-05-08: For approved-deferral hardening fixtures, keep the scenario truthful by pairing `quality\hardening-gate.md` with a canonical fixture-local `.squad\decisions.md` entry; otherwise the case only models a claimed defer, not an actually approved one.
- 2026-05-08: When extending an existing governance regression suite with future-phase scenarios, fork from the suite's passing fixture baseline and vary only the new gate artifact plus approval evidence. That keeps the later test signal on the intended gate instead of reintroducing unrelated baseline failures.
- 2026-05-08: Hardening-gate contract coverage should assert three bounded states separately—`blocked`, `deferred-with-approval`, and `ready`—and prove them through the shared parser/approval helpers so fixture drift is caught before `run-hardening-gate.ps1` or `validate-governance.ps1` starts depending on the same contract.
- 2026-05-08: When validator enforcement starts depending on a later-phase hardening artifact, bind the check to the concrete Phase 2 planning metadata and the shared hardening-state helper together. That keeps verdict names, artifact paths, and human-approval semantics aligned across scaffolding, orchestration, fixtures, and fail-closed governance.
- 2026-05-08: When an iteration introduces a new fail-closed gate, require that same iteration to pass the live gate under accepted review before moving to retro. A self-approved defer would erase the dogfood signal and weaken closure evidence.
- 2026-05-08: Infrastructure-only iteration slices require hardening-gate review for the bounded scope actually delivered, not for deferred User Story work. Use the canonical v1 schema (five concern rows: security-surface, error-handling-expectations, retry-idempotency-requirements, test-integrity-targets, operational-resilience-concerns) and mark each concern honestly as `addressed` or `not-applicable` based on what the current slice creates, without pretending story-specific routing behavior already exists.
- 2026-05-10: When integration tests for handoff-visibility requirements verify only fixture files directly (not the actual scaffold or replay script output), they prove fixture content but not that the scripts surface that content. For FR-011-class "user-facing handoff" requirements, always trace at least one test path through the full scaffold/replay invocation against the fixture to confirm the generated output carries the required fields — not just that the fixture state.md contains them.
- 2026-05-10: A task that names three files in its description (e.g., "Surface X in script-A.ps1, script-B.ps1, and routing.md") is only complete when all three are updated. Updating one file satisfies the requirement for that file only; the other two remain open. Verify deliverable completeness by checking each named file's diff before marking the task done.
- 2026-05-10: When a foundational task (T007) and a user-story task (T019) both target the same script for different layers of the same signal (generic escalation vs. cap-specific fields), make the distinction explicit in the task description. If the foundational task covers generic signal surface and the story task adds cap-specific fields, the review must verify the cap-specific additions independently — the generic signal passing does not imply cap-specific passing.
- 2026-05-11: Approval-recording boundary audit revealed four recurring defect patterns: (1) state.md status lags behind hardening-gate signed verdict, (2) plan.md conflates planning-level approval with hardening-gate-triggered implementation authorization instead of recording both separately, (3) hardening-gate.md Approval Ref field left blank, violating traceability mandates, (4) status summaries undercount feature-specific concerns in the Concern Review table. These gaps point to a need for stricter artifact-truth validation before sign-off: hardening-gate.md Overall Verdict change requires cascading status updates to state.md and plan.md, and Approval Ref must be populated before gate sign-off is recorded as complete.
- 2026-05-11: Re-audit of Iteration 005 approval-recording boundary repair confirms all four defects are corrected and verified by governance validation script. Pattern observed: each human-directed exception to a governance trap (e.g., `Approval Ref: —` override) requires explicit documented rationale in the decision inbox so that exception remains auditable and does not become precedent without re-decision. The repairs establish a truthful artifact set ready for implementation: state.md status matches hardening-gate verdict, plan.md records both planning-level approval (2026-05-10) and implementation authorization (2026-05-11) distinctly, hardening-gate.md concern count is accurate (six polish-specific), and Approval Ref exception is documented.
- 2026-05-11: When a later polish slice narrows an earlier validation command list, review must verify three separate truths before acceptance: the approved artifacts now name the reduced lane exactly, the rerun evidence stays on that lane only, and any user-facing example text is replay-checked against live scaffold/review output rather than fixture files alone.
- 2026-05-11: Retrospective audits must verify that friction events (rejected boundaries, repair cycles, rework authorization) are named explicitly in the "What Didn't Go Well" or equivalent friction section, even when friction is subsequently resolved. Retrospectives that move repair narratives into "What Went Well" while claiming zero friction in the negative section are artifact smoothing and fail truthfulness audit. A rejected approval boundary plus an independent repair cycle in the same iteration constitutes friction that must be named; the retrospective may explain how it was resolved, but omitting the friction event itself from the negative section is incomplete truth.
- 2026-05-12: Feature 012 iteration 002 acceptance confirms four enduring review patterns: (1) Replay tests must use the real governance validator path and assert on user-visible output (status, findings, summary) rather than internal state alone; (2) Known-traps corpus rows must be seeded and cited in the review boundary and preserved regression lanes must stay aligned together; (3) Approval-reuse detection requires whitespace normalization + markdown-emphasis stripping only, with false-positive protection via distinct-quote validation, and structured FAIL output naming both iterations; (4) Bookkeeping-only classifier changes (e.g., helper-reuse in specrew-start.ps1) must stay additive with no validator CLI changes and must pass both fixture + live tree regression. These patterns apply to future approval-reuse, over-claim, and hardening-gate enforcement work.
- 2026-05-11: Re-audit of Iteration 005 retrospective repair confirms that the three-part truthfulness boundary (no rejection/repair smoothing, accurate Approval Ref traceability, consistent state.md status fields) is satisfied when: (1) a dedicated "Friction Encountered and Resolved" section isolates the rejection/repair event from remediation narratives, (2) Approval Ref exceptions are documented in decision-inbox entries with governance rationale, and (3) state.md timestamps record both the repair trigger event and the authorization moment distinctly. When all three are present, the retrospective tells a coherent, auditable story without invention of re-review blockers.
- 2026-05-11: Feature 007 iteration 001 Foundation & Governance review establishes that pure documentation-and-governance slices (no runtime code, only prompt/template/agent-guidance changes) can satisfy all FR/US requirements through planning-time evidence alone when: (1) artifacts demonstrate explicit handoff-semantic understanding (progress status + next step) with concrete examples, (2) governance-acronym trap detection rules are absorbed into coordinator guidance before soft-validator implementation, (3) Squad.agent.md changes are paired with explicit session-restart warnings, and (4) runtime validation is explicitly deferred to a subsequent iteration with clear implementation targets. This pattern keeps Foundation slices honest about what they deliver (guidance and design) versus what they defer (runtime enforcement and integration tests).
- 2026-05-12: When preparing a review rubric from a hardening gate, bind each blocking concern to exact proving commands, artifact paths, and false-positive/false-negative checks before implementation finishes. For validator-governance work, "implemented", "enforced", "observable", "documented", and "regression-safe" are separate acceptance lenses; collapsing them into one story invites soft acceptance.
- 2026-05-12: For validator hardening slices that span `validate-governance.ps1` and `specrew-start.ps1`, do not accept on helper fixtures alone. Require the current-iteration replay harness, the prior-iteration replay harness, the restart-flow regression suite, and repo-wide `validate-governance.ps1` together so additive behavior is proven across both governance and restart-policy surfaces.
- 2026-05-12: Once `review.md` is scaffolded for a live iteration, treat it as lifecycle state, not just documentation. `plan.md` must move from `executing` to `reviewing`, and `state.md` must stop saying review is deferred, because `validate-governance.ps1` rejects stale executing/pending-review narratives as soon as the review artifact exists.
- 2026-05-12: Feature 014 handoff-format-scoping review boundary confirms that soft-warning governance rules (non-blocking detector rules with escalation guidance) pass all five acceptance lenses when the rule implementation is additive to prior rules, existing regression suites stay green, and reviewer guidance surfaces (review.md verdict narrative, hardening-gate.md Concern Review) model the new format correctly. Dogfooding the new format rules inside the review-boundary verdict itself (not just asserting fixture output) provides the strongest confidence in observable coherence.
- 2026-05-13: Release-truth reviews should verify annotated tags with peeled refs locally and on `origin` (`refs/tags/<tag>^{}`), because the tag-object SHA is not the release anchor. For public-readiness slices, pair that tag check with clean-fixture, drift-fixture, pre-feature live-iteration, and repo-wide validator runs so additive warning behavior is proven without mistaking repo-wide green for warning-lane coverage.
- 2026-05-13: Feature 015 public-readiness review boundary confirms that version-management rule enforcement and versioning-doc alignment across config.yml, README.md, docs/versioning.md, CHANGELOG.md, spec, and release tags (v0.13.0 -> 21d9e7f, v0.14.0 -> 3ff32d4) stays additive. Public-readiness validator lane passes clean fixture without warnings, drift fixture with expected warnings, pre-015 iterations unchanged, and repo-wide validator green. Retrospective authorization deferred pending human sign-off.
