## Learnings

**Lesson:** Feature 023 Iteration 001 achieved zero-variance delivery (14.5 SP planned = 14.5 SP actual) and 100% first-review acceptance despite touching 7+ file locations and managing a complex reader audit. Success factors: (1) Spec provided clear v0/v1 dispatch semantics before planning, preventing scope re-interpretation; (2) State Reader Audit (T003) front-loaded a fixed reader inventory (13 readers across 5 scripts), blocking scope creep; (3) Legacy Fixture Corpus strategy (hand-curated versions + current-version addition) was explicit in spec, not discovered during implementation; (4) Bootstrap principle dogfooding (Feature 023's own writers/readers/fixtures demonstrate the pattern) earned credibility without extra validation cycles. For future infrastructure features, replicate this pattern: clear dispatch semantics in spec, reader/writer inventory audit in Phase 1, explicit fixture strategy, internal dogfooding validation. **(Feature 023 Iteration 001 Retrospective — 2026-05-19)**

**Lesson:** Bootstrap principle dogfooding is the strongest internal validation when a feature requires downstream adoption (schema v1 readers, hashtable-based parsing, fixture corpus updates). Feature 023 Iteration 001 demonstrated the pattern internally: (1) state writers emit the v1 markers they require readers to accept; (2) state readers use the hashtable migration discipline they enforce; (3) fixture corpus includes the feature's own 0.23.0 boundary; (4) regression suite exercises the feature's own surfaces. This self-consistent pattern closed credibility gaps with reviewers and validators. For future schema/governance features, formalize bootstrap dogfooding as a required audit lens: verify writers/readers/fixtures are self-aligned before requesting review. **(Feature 023 Iteration 001 Retrospective — 2026-05-19)**

**Lesson:** Retro-boundary bookkeeping repair distinguished stable retro-boundary-completion commit from later learning-capture commit. Feature 023 Iteration 001 showed that the retro-boundary should be marked at the commit where the retrospective artifact (retro.md) itself is created and ready for signoff (commit 74e0f40), not deferred until a later "learning record" commit. The pattern: (1) retro.md scaffolded and populated with findings; (2) state.md updated to post-retro status; (3) decisions.md records actual commit reference (not "pending"); (4) single commit marks the boundary. Subsequent learning-capture commits (e.g., appending to retro-facilitator history.md) are administrative/governance noise, not boundary transitions. Rule for future retro work: Populate retro.md, update state.md, fix decisions.md commit reference, commit all three together as one retro-boundary commit. Do not leave decisions.md with "(pending commit)" or retro.md with "(populated after commit)" placeholders. **(Feature 023 Iteration 001 Retro-Boundary Repair — 2026-05-19)**
