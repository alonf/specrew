## Learnings

**Lesson:** Perfect estimation accuracy across documentation-heavy iterations requires clear task boundaries and stable scope at the start. Iteration 001 delivered all 10 story points at estimated effort with zero variance. This success reflects (1) well-defined coordination scope in planning (T001-T006 are all guidance and documentation), (2) no discovery surprises (handoff contract semantics were stable pre-implementation), (3) honest task decomposition (each task is discrete and has clear acceptance criteria). When starting a new Spec Kit feature with high coordination burden, invest extra planning rigor upfront so task boundaries remain clear throughout execution. **(Feature 007 Iteration 001)**

**Lesson:** Plain-language-first principle must be absorbed into multiple guidance artifacts for durability across session boundaries. Iteration 001 embedded the governance-acronym trap detection rule (three-or-more governance acronyms in lead without paraphrase) into four separate artifacts: T001 coordinator prompt (plain-language guardrail + examples), T002 handoff template (concrete patterns), T005 governance checklist (soft-warning check), and T006 soft-validator design (pseudo-code). This redundancy ensures that even if one artifact gets skipped or outdated, the principle survives in the others. Single-artifact embedding (e.g., just the checklist) is fragile. Multi-artifact redundancy is the durability pattern. **(Feature 007 Iteration 001)**

**Lesson:** Startup-loaded configuration boundaries (like `.github/agents/squad.agent.md` changes) require explicit session-restart discipline and must be documented at every editing point. T004 added the session-restart requirement to Squad.agent.md itself, making it self-documenting. However, the team running Iteration 002 planning must still be told explicitly: "Do not try to continue in the current session. End this session. Start a new session." This awareness must be carried in the retrospective handoff note, not inferred from documentation. Formalize this as a known coordination trap: startup-loaded config changes require synchronous session-boundary commit, not just documentation. **(Feature 007 Iteration 001)**

**Lesson:** Soft-validator design (T006) providing clear implementation target without premature runtime complexity shows strong boundary discipline. Iteration 001 stayed purely in documentation/guidance (Phase 1 + Phase 2) and provided Iteration 002 with an explicit design contract (T006: detection rule, pseudo-code, integration points, five test scenarios). This deferred complexity pattern allows Iteration 001 review to focus on handoff-semantics correctness, not validator logic. Iteration 002 inherits a clean implementation target. This is the Foundation & Governance pattern to replicate: deliver design, defer implementation, provide clear contract. **(Feature 007 Iteration 001)**

**Lesson:** Zero-rework success (all six tasks accepted on first review submission) reflects careful pre-review hardening by the implementation team. Iteration 001 did not have late-found gaps, spec misalignments, or governance-integration surprises. This cleanliness suggests the coordinator guidance and handoff semantics were well-understood before review started. In future Foundation & Governance iterations, replicate this by ensuring a "before-review checkpoint" (like the one recommended for Iteration 002 planning) to validate concept understanding before formal review. **(Feature 007 Iteration 001)**
