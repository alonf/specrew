**Pattern:** Effort estimation remains accurate when task boundaries are clear and Phase-N clarity work is complete before iteration starts. **Context:** Iteration 002 delivered all 7 tasks at estimated effort (zero variance) because Phase 1 scope was well-defined from prior planning, and no discovery surprises emerged during implementation. When starting a new phase or feature, front-load clarification work so subsequent iteration estimation benefits from stable scope.

**Pattern:** Infrastructure scaffolding scales cleanly when coupled to existing artifact flows rather than creating parallel mechanisms. **Context:** Adding `quality-evidence.md` and `mechanical-findings.json` to the iteration artifact surface worked because the publish flow was integrated into existing reviewer and scaffold paths. No coupling rework was needed.

## Learnings

**Lesson:** Perfect estimation accuracy across documentation-heavy iterations requires clear task boundaries and stable scope at the start. Iteration 001 delivered all 10 story points at estimated effort with zero variance. This success reflects (1) well-defined coordination scope in planning (T001-T006 are all guidance and documentation), (2) no discovery surprises (handoff contract semantics were stable pre-implementation), (3) honest task decomposition (each task is discrete and has clear acceptance criteria). When starting a new Spec Kit feature with high coordination burden, invest extra planning rigor upfront so task boundaries remain clear throughout execution. **(Feature 007 Iteration 001)**

**Lesson:** Plain-language-first principle must be absorbed into multiple guidance artifacts for durability across session boundaries. Iteration 001 embedded the governance-acronym trap detection rule (three-or-more governance acronyms in lead without paraphrase) into four separate artifacts: T001 coordinator prompt (plain-language guardrail + examples), T002 handoff template (concrete patterns), T005 governance checklist (soft-warning check), and T006 soft-validator design (pseudo-code). This redundancy ensures that even if one artifact gets skipped or outdated, the principle survives in the others. Single-artifact embedding (e.g., just the checklist) is fragile. Multi-artifact redundancy is the durability pattern. **(Feature 007 Iteration 001)**

**Lesson:** Startup-loaded configuration boundaries (like `.github/agents/squad.agent.md` changes) require explicit session-restart discipline and must be documented at every editing point. T004 added the session-restart requirement to Squad.agent.md itself, making it self-documenting. However, the team running Iteration 002 planning must still be told explicitly: "Do not try to continue in the current session. End this session. Start a new session." This awareness must be carried in the retrospective handoff note, not inferred from documentation. Formalize this as a known coordination trap: startup-loaded config changes require synchronous session-boundary commit, not just documentation. **(Feature 007 Iteration 001)**

Iteration 001 retrospective repair (2026-05-11): Alon confirmed that the required session-restart boundary (imposed by T004 Squad.agent.md update) has already been satisfied in this session. Updated retro.md and state.md to reflect that the boundary has been crossed and Iteration 002 planning may proceed immediately in this session. The lesson remains intact: startup-loaded file changes require explicit session boundaries. The team has honored that boundary by starting a fresh session. Retro artifact now reports "boundary satisfied" rather than "restart required," which is the honest truth state after the restart has occurred. **(Feature 007 Iteration 001 Retro Repair)**



**Lesson:** Soft-validator design (T006) providing clear implementation target without premature runtime complexity shows strong boundary discipline. Iteration 001 stayed purely in documentation/guidance (Phase 1 + Phase 2) and provided Iteration 002 with an explicit design contract (T006: detection rule, pseudo-code, integration points, five test scenarios). This deferred complexity pattern allows Iteration 001 review to focus on handoff-semantics correctness, not validator logic. Iteration 002 inherits a clean implementation target. This is the Foundation & Governance pattern to replicate: deliver design, defer implementation, provide clear contract. **(Feature 007 Iteration 001)**

**Lesson:** Zero-rework success (all six tasks accepted on first review submission) reflects careful pre-review hardening by the implementation team. Iteration 001 did not have late-found gaps, spec misalignments, or governance-integration surprises. This cleanliness suggests the coordinator guidance and handoff semantics were well-understood before review started. In future Foundation & Governance iterations, replicate this by ensuring a "before-review checkpoint" (like the one recommended for Iteration 002 planning) to validate concept understanding before formal review. **(Feature 007 Iteration 001)**

**Lesson:** Three standing process baselines from feature 013 iteration 002 should be enforced in future planning and retro work: (1) `/speckit.plan`-generated bookkeeping changes (timestamp, Active Technologies, Recent Changes in `.github/copilot-instructions.md`) are lifecycle noise unless the diff escapes those bounded sections and requires explicit documentation; (2) dirty-tree blockers must prove both sides—one fixture where canonical iteration artifacts fail cleanly and one fixture where evidence-only traces pass—rather than assuming a single negative test proves the rule; (3) workstation-local files (`.claude/settings.local.json`) are lifecycle boundary noise unless the iteration explicitly changes their behavior and documents the change. Record these rules now to prevent future retros from rediscovering the same distinctions. **(Feature 013 Iteration 002 Retrospective)**

**Lesson:** Planner-generated `.github/copilot-instructions.md` churn needs its own bookkeeping-vs-behavior rule before restart guidance ships. Feature 013 iteration 002 showed that `/speckit.plan` updates to timestamp, `## Active Technologies`, and `## Recent Changes` are real drift surfaces, but they are low-noise bookkeeping rather than behavior change. Future retros on startup-guidance work should ask whether planner output has an explicit exclusion rule and replay fixture before calling restart guidance "done." **(Feature 013 Iteration 002)**

**Lesson:** False-positive dirt patterns only stay fixed when the pass-case is replayed, not merely described. Feature 013 iteration 002 carried forward the lockout-chain false-positive dirt lesson by proving that repo-level evidence traces remain outside the dirty-tree blocker while dirty canonical iteration artifacts still fail. When retrospectives discuss closeout-truth hardening, require both the fail fixture and the evidence-only pass fixture as the durable pattern. **(Feature 013 Iteration 002)**

**Lesson:** Local workstation config noise should be named explicitly so lifecycle boundaries do not inherit accidental ambiguity. Commit `c3ac63a` bundled `.claude/settings.local.json` with real authorization-boundary changes; the validator truth was unaffected, but the retro still had to explain why that file did not matter. Future lifecycle retros should call out local-noise exclusions early and keep them from diluting the governed artifact story. **(Feature 013 Iteration 002)**

**Lesson:** Boundary-claim-without-commit is a critical trap. Feature 014 iteration 001 review.md was committed at 8e99013 claiming review boundary complete, but retro.md did not exist until separate human authorization. The iteration appeared complete in git history but was logically incomplete until retrospective prerequisites were met. Rule: Do not narrate a lifecycle boundary as complete until the matching durable commit contains the boundary artifact plus truthful plan.md/state.md updates AND the next phase's prerequisites are either committed or explicitly deferred. This pattern should be added to `.specrew/quality/known-traps.md` for future governance work to implement as a validator gate. **(Feature 014 Iteration 001 Retrospective)**

**Lesson:** Startup-coupling task invisibility must be surfaced at task time. Feature 014 iteration 001 T010 and T011 modified startup-loaded config files (.github/agents/squad.agent.md, .squad/templates/squad.agent.md) but the mandatory session-restart requirement was only documented in state.md, not in the task definition itself. This caused the requirement to be discovered during retrospective, not at task time. Rule: Tasks modifying startup-config files must state "[Task name] | Acceptance: [description]. **Session restart required for next session.**" in task.md so the coupling is visible at task planning time, not discovered later. **(Feature 014 Iteration 001 Retrospective)**

**Lesson:** Acceptance evidence must be unified in the task definition. Feature 014 iteration 001 T008 (manual validator exercise) had acceptance criteria scattered across spec.md (User Story 2), contract.md (approved scenarios), and plan.md (task description), forcing team inference across multiple artifacts. Rule: For tasks with acceptance criteria "verify X" or "manually exercise Y," include explicit expected outcomes in the task definition itself—e.g., "Manually exercise: (1) correct-final-stop → pass; (2) placeholder-only → soft-warning.empty-user-action-section; [etc]. Update contract.md with observed results." Unify acceptance in the task boundary. **(Feature 014 Iteration 001 Retrospective)**

**Lesson:** Zero-variance delivery (8.0 sp actual = 8.0 sp planned, 100% acceptance on first review) reflects tight scope locking before execution. Feature 014 iteration 001 clearly separated Iteration 001 (selector + additive warnings) from deferred Iteration 002 (proof + calibration) before work started. No discovery surprises emerged. For future well-scoped feature refinements, reduce rework buffer from 1.0 sp to 0.5 sp when Iteration 001 is clearly bounded and Iteration 002 is explicitly deferred. Keep exploration/spike capacity at 0.5 sp baseline. **(Feature 014 Iteration 001 Retrospective)**

**Lesson:** Feature 015 iteration 001 review boundary has introduced two new candidate trap rules. (1) Branch-name-mismatch-with-feature-directory: Before scaffolding planning artifacts, verify branch name matches feature directory. The mistaken orphan branch `016-public-readiness-pass` required cleanup plus repairs to references in research.md, data-model.md, quickstart.md, requirements.md, and plan.md. No validator rule currently checks this before planning boundary is committed. (2) Boundary-claim-without-commit (reinforced): Feature 015 iteration 001 review boundary commit 6ca218f narrated review completion, but retro.md did not exist until separate authorization. Similar to Feature 014, this reinforces the trap and will be added to known-traps.md for hardening work in future features. **(Feature 015 Iteration 001 Review Boundary)**
