## What this review covers

Slice 9 of Proposal 108 added per-host Crew runtime install — closing the architectural gap the user observed in their `specrew start --host claude` session. Before Slice 9, only Copilot+Squad had the 5-agent Crew (Spec Steward, Planner, Implementer, Reviewer, Retro Facilitator) deployed. After Slice 9, all 4 hosts (Copilot, Claude, Codex, Antigravity) get the same baseline team, translated to each host's native subagent format.

This document reviews **what was implemented**, **what was deferred**, **what changed in design vs Proposal 108 as drafted**, and **what the user should verify in manual smoke tests**.

## What shipped

### 1. Single source of truth: `.specrew/team/`

Decision (locked by user via `AskUserQuestion` at design time): the canonical team location is `.specrew/team/agents/<role>.md`. Host-native locations are GENERATED VIEWS — regenerated on every `specrew start --host <kind>`.

```text
.specrew/team/agents/           ← SINGLE SOURCE OF TRUTH (user edits here)
├── spec-steward.md             (5 baseline charters auto-seeded from
├── planner.md                    extensions/specrew-speckit/squad-templates/agents/
├── implementer.md                on greenfield `specrew init`)
├── reviewer.md
├── retro-facilitator.md
└── <user-added>.md             ← e.g., security-analyst.md (Phase 10 follow-up)

.squad/agents/<role>/charter.md ← GENERATED by Install-CopilotCrewRuntime
.claude/agents/<role>.md        ← GENERATED by Install-ClaudeCrewRuntime
.codex/agents/<role>.toml       ← GENERATED by Install-CodexCrewRuntime
.agents/agents/<role>.md        ← GENERATED by Install-AntigravityCrewRuntime
```

**Why every-launch translation:** drift between canonical and host-native becomes structurally impossible because translation runs every `specrew start`. Cost: ~50ms for 5-7 file translations. Worth the architectural cleanness.

### 2. 5th contract function

`hosts/_contract.md` + `hosts/_registry.ps1` extended with one row:

```powershell
'InstallCrewRuntime' = 'Install-{0}CrewRuntime'
```

Each host's `handlers.ps1` implements the body. Signature uniform across hosts:

```powershell
function Install-<Kind>CrewRuntime {
    param([string]$ProjectPath, [switch]$DryRun)
    # 1. Iterate canonical roles via Get-SpecrewCanonicalAgentRoles
    # 2. For each role: read .specrew/team/agents/<role>.md
    # 3. Translate to host-native format
    # 4. Write to host's AgentDir (from manifest)
    # 5. Return pscustomobject @{ Actions[]; CrewRuntimePath; Notices[] }
}
```

## What changed in design vs Proposal 108 as drafted

Proposal 108 originally deferred the canonical-vs-eager decision (Proposal 104 Question 5). The user forced the resolution during implementation by asking "do we create teams for each host?" That triggered the design-question flow:

1. **Canonical path**: `.specrew/team/` chosen over `.specrew/coordinator/` or `.specrew/crew/`. Decision: keeps `.specrew/coordinator/` reserved for the coordinator-prompt overlay (Slice 10 candidate); "team" is the user-facing term for the 5-agent collective.

2. **Squad path**: `.squad/agents/<role>/charter.md` remains Squad CLI's read location. `Install-CopilotCrewRuntime` translates from canonical → Squad's expected path (rather than migrating Squad to read from `.specrew/team/`). Decision: avoid Squad CLI compatibility risk; Squad doesn't control its read path.

3. **Translation timing**: every `specrew start --host <kind>` invocation. Decision: always-in-sync; drift impossible; cost (~50ms) is acceptable.

These three decisions locked at implementation time and are now baked into the architecture.

## What was deferred (Slice 9 follow-ups)

1. **`specrew team add/update/remove` rewiring** — today these commands still write to `.squad/team.md` (where Squad CLI roster lives). They need to ALSO write canonical charters to `.specrew/team/agents/<name>.md`. Separate slice (touches `scripts/specrew-team.ps1`).

2. **Coordinator overlay translation** — each host has a top-level coordinator file (`.github/agents/squad.agent.md` for Copilot, `CLAUDE.md` for Claude, `AGENTS.md` for Codex/Antigravity). The coordinator-prompt body is currently generated by `deploy-squad-runtime.ps1` for Squad/Copilot; the other hosts get a thinner version via `coordinator-prompt-surgery.ps1`. A future slice should canonicalize this at `.specrew/coordinator/specrew-governance.md` and translate per host the same way the team charters now flow. Slice 10 candidate.

3. **`.specrew/team/ceremonies/`** + `decisions.md` migration — Squad's `.squad/ceremonies.md` and `.squad/decisions.md` are still per-host (Squad-specific). Future slice migrates these to canonical too. Out of scope for Slice 9.

4. **Antigravity body real-launch validation** — Antigravity manifest declares it `Status: supported` after Slice 9, but the per-host format is medium-confidence. The user should smoke-test on actual `agy` CLI and surface any deviations.

5. **Codex coordinator delegation semantics** — Codex has no user-defined slash-command surface (FR-013 from F-040). So delegating to specialists on Codex happens via prose ("Implementer: write the code") rather than `/agent <role>`. The agent files are in place; the orchestration prose is part of the coordinator-overlay slice deferred above.

## Lessons learned (folded into the architecture-docs Risks section)

Two PowerShell parsing gotchas surfaced during implementation. Both are subtle, both bit me, both are now documented:

1. **Comma-bound-tighter-than-plus inside array literals**

   ```powershell
   # WRONG — parses as one big concat:
   $arr = @(
       'a' + $role + '.md',
       'b' + $role + '.md'
   )
   # Result: $arr.Count == 1; $arr[0] == 'a' + $role + ('.md', 'b') + $role + '.md'

   # CORRECT — parenthesize each element OR use -f:
   $arr = @(
       ('a{0}.md' -f $role),
       ('b{0}.md' -f $role)
   )
   ```

2. **`@(...) -join "`n"` on single expression returns ZERO LFs**

   ```powershell
   # WRONG:
   $x = @('a', 'b', 'c') -join "`n"
   # Result: $x has 0 LF chars; all elements concatenated with no separator

   # CORRECT — two-step:
   $lines = @('a', 'b', 'c')
   $x = $lines -join "`n"
   ```

Both gotchas exist because the PowerShell parser ties `,` to the surrounding expression context tighter than expected. Any contributor extending the host packages should know these. They are now in the [host-package architecture's Risks section](../architecture/host-package-architecture.md) and the [add-a-new-host guide's Common Pitfalls](../how-to/add-a-new-host.md).
