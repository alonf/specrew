## Problem Statement

Specrew's core value depends on the interaction model between the human developer and the Squad+Specrew lifecycle. The developer provides intent and approves boundaries, Squad advances the lifecycle, artifacts capture the full substance, and the console output serves as the primary user interface for inspection and decision-making.

This interaction model currently breaks down in three coupled ways:

1. **Boundary Discipline**: Squad can advance across multiple lifecycle boundaries without explicit per-boundary human authorization.
2. **Essence-in-Console**: Boundary stops can be procedurally correct but substantively thin, forcing the user to open artifacts just to understand what happened.
3. **Click-Through Navigation**: Artifact references can appear as bare paths rather than `file:///` links, preventing immediate drill-down from the console.

These pillars must hold together. A stop without substance is weak, substance without clickable navigation is inefficient, and both are irrelevant if Squad skips the stop entirely. This feature codifies the three-pillar interaction model so Specrew reliably stops at the right boundaries, explains the substantive state in the console, and provides clickable artifact links for deeper inspection.

## Clarifications

### Session 2026-05-13

- Q: What does "continue" authorize? → A: Single-boundary advance only; advance to the next single boundary stop, then halt and ask again.
- Q: What substantive threshold applies to "What I just did"? → A: Strict AND (`>=3` identifiers AND `>=50` words) for planning, implementation, review, and retro; soft OR (either satisfies) for iteration-closeout and feature-closeout; not tunable per project.
- Q: What rollout severity applies to bare paths in boundary handoffs? → A: Soft-warning in Iteration 1, then promote to hard-fail in Iteration 2 after exemption-list integration tests prove bounded false positives; implement severity as a config flip, not a rewrite.
- Q: How granular should the actionable-request warning be? → A: Emit one warning per handoff, but name each missing component: boundary-name, inspection-target, and verdict-required.
- Q: How are paired authorizations recorded? → A: As two distinct `.squad/decisions.md` entries auto-generated by Squad from the user's single authorization paste.
- Q: Do Pillar 2 soft warnings graduate in Feature 016? → A: No; they remain soft-warning throughout Feature 016, with graduation candidacy noted in corpus cross-references for future consideration.
- Q: How are authorization entries created? → A: Squad auto-generates them from the user's authorization text, parsing the block and capturing boundary, ISO 8601 UTC timestamp, commit reference, and verbatim authorization phrase; the user can review or override during boundary inspection.
- Q: Can projects extend the bare-path exemption list? → A: Yes, via `.specrew/config.yml`, but only with human approval recorded alongside approver name and rationale.
- Q: What text does the validator inspect? → A: Squad-authored artifacts and handoffs only; it does not parse conversation transcripts or user-typed text.
- Q: Do substantive-content rules extend to artifacts in Feature 016? → A: No; Feature 016 covers console handoffs only, while a passive `thin-artifact-content` corpus row records future graduation candidacy.

## Scope Boundaries

### In Scope

- Updating the coordinator prompt surfaces to require per-boundary stop-and-ask behavior and to enumerate the canonical lifecycle boundaries by name.
- Forbidding bundled boundary advances, including cases where the user types "continue" or other broad multi-step language.
- Auto-generating `.squad/decisions.md` authorization entries from user authorization pastes, including paired-entry expansion when one paste authorizes two adjacent boundaries.
- Defining validator rules that detect missing intervening human authorization between boundary commits.
- Requiring substantive three-section boundary handoffs that surface concrete decisions, changes, risks, and next actions in the console.
- Explicitly scoping the validator to Squad-authored handoffs/artifacts and excluding user-authored transcript text from rule evaluation.
- Requiring `file:///` URL format for artifact references in Squad narration and boundary handoffs.
- Defining validator rules, approved exemption extensions, corpus rows, integration tests, README guidance, and handoff-template examples needed to enforce the three-pillar interaction model.

### Out of Scope

- Mermaid or PlantUML diagrams in artifacts; that visual layer is deferred to Feature 017.
- Real-time interruption of in-flight tool calls; this feature governs commit and lifecycle boundaries, not tool-call boundaries.
- Multi-host runtime support; the scope targets the current Copilot+Squad host model.
- Changing the established lifecycle phases, artifact contract, or boundary commit signature family introduced by prior features.
- Introducing boundary bypasses for "trivial" iterations; all boundaries still apply.
- GUI changes; the interaction model remains console-first.
- Treating implicit consent or prior conversational context as sufficient boundary authorization.
- Applying Pillar 2 substantive-content validation to `retro.md`, `review.md`, hardening-gate evidence sections, or other closed artifacts; Feature 016 governs console handoffs only.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Enforce one authorization per boundary (Priority: P1)

A Specrew user can rely on Squad to stop at each lifecycle boundary and request explicit human authorization before advancing to the next boundary.

**Why this priority**: Boundary discipline is the foundation of the interaction model. If Squad advances without approval, the rest of the experience loses its control surface.

**Independent Test**: Replay a lifecycle sequence that previously bundled multiple boundaries and confirm Squad now stops after each boundary commit and requests fresh authorization before continuing.

**Acceptance Scenarios**:

1. **Given** an iteration reaches a planning boundary, **When** Squad records that boundary commit, **Then** Squad stops and asks for the next explicit authorization before entering the next boundary.
2. **Given** the user replies with "continue", **When** Squad resumes, **Then** it advances to the next single boundary stop and halts again rather than traversing multiple boundaries.
3. **Given** two boundary commits appear without an intervening authorization entry, **When** governance validation runs, **Then** the validator emits a structured FAIL naming the offending gap.
4. **Given** the user pastes a single authorization that covers hardening-gate sign-off and implementation authorization, **When** Squad records it, **Then** Squad creates two distinct `.squad/decisions.md` entries from that single paste and allows the user to review or override them during boundary inspection.

---

### User Story 2 - Understand the substance from the console alone (Priority: P1)

A Specrew user can read a boundary handoff in the console and understand what was done, why Squad stopped, what risks or decisions matter, and what specific review or authorization is needed next.

**Why this priority**: The console is the first interface. If it lacks substantive information, the user must manually mine artifacts before they can make a decision.

**Independent Test**: Present a fresh-context reviewer with only a boundary handoff and confirm they can summarize the completed work, the active boundary, and the required next decision without opening artifacts first.

**Acceptance Scenarios**:

1. **Given** Squad stops at planning, implementation, review, or retro, **When** it renders the handoff, **Then** the "What I just did" section includes at least 3 specific identifiers AND at least 50 words; for iteration-closeout and feature-closeout, either threshold alone is sufficient.
2. **Given** Squad explains why it stopped, **When** the user reads the message, **Then** the specific boundary phase being entered is named explicitly rather than implied generically.
3. **Given** Squad requests human action, **When** the user reads the request, **Then** the message names the exact boundary, inspection targets, and verdict required, and any warning emitted for omissions is a single warning that names every missing component.

---

### User Story 3 - Navigate directly from handoffs into artifacts (Priority: P1)

A Specrew user can click artifact references in Squad's narration and boundary handoffs to open the relevant files immediately for deep-dive inspection.

**Why this priority**: Click-through navigation turns the console into a practical control surface rather than a dead-end summary.

**Independent Test**: Inspect boundary handoffs and ordinary narration and confirm artifact references use `file:///` links, while exempt path contexts do not trigger false alarms.

**Acceptance Scenarios**:

1. **Given** Squad cites an artifact in a boundary handoff, **When** the user inspects the message, **Then** the citation uses `file:///` URL format rather than a bare path.
2. **Given** Squad includes a bare path in a boundary handoff, **When** governance validation runs during Iteration 1, **Then** the validator emits a soft warning for the navigation violation, and **When** the same rule is promoted in Iteration 2 after exemption-list integration tests pass, **Then** it emits a hard FAIL.
3. **Given** a path appears inside an exempt context such as inline code, JSON, YAML, regex, shell-command arguments, or a human-approved project-specific extension, **When** governance validation runs, **Then** the validator does not flag that occurrence as a navigation error.

---

### Edge Cases

- A user types "continue" expecting a single-step advance, but Squad must still treat it as one-boundary authorization rather than blanket permission.
- Paired hardening-gate sign-off and implementation authorization must still be recorded as distinct authorizations even when approved in close succession.
- A handoff may contain enough words but still fail substance if it lacks concrete identifiers that anchor the summary to actual work.
- Iteration-closeout and feature-closeout handoffs may be brief, but they still must satisfy at least one substantive threshold so the stop is not empty.
- A handoff may contain valid `file:///` links but still fail if the referenced files do not exist.
- Bare paths inside shell commands, code blocks, structured literals, regex patterns, glob arguments, or approved project-specific exemption contexts must remain exempt to avoid noisy false positives.
- The validator must not inspect user-authored transcript text or pasted authorization prose directly; only Squad-authored outputs and generated artifacts are in scope.
- Auto-generated authorization entries must preserve the user's verbatim authorization phrase while still allowing the user to override metadata during boundary inspection.
- Legacy iterations created before Feature 016 must remain valid under grandfathering rules rather than being retroactively invalidated.

## Requirements *(mandatory)*

### Functional Requirements

#### Pillar 1: Boundary Discipline

- **FR-001**: The coordinator prompt MUST require Squad to stop and request explicit human authorization after EACH of the 7 iteration boundary commits: planning, hardening-gate-and-implementation-auth, implementation, review-boundary, review-verdict-signoff, retro-boundary, and iteration-closeout. These 7 iteration boundaries MUST be enumerated by name in the coordinator prompt. `feature-closeout` remains a separate feature-level boundary and is not part of the per-iteration count. **Owner role**: Governance steward. **Delivery window**: Iteration 1.
- **FR-002**: The coordinator prompt MUST explicitly forbid bundled boundary advances. The rule: "Each boundary commit requires its own immediately-preceding authorization; one human authorization advances at most one boundary." Even when the user says "continue" or provides broad multi-step authorization, Squad MUST NOT emit multiple boundary commits without intervening explicit per-boundary authorization. **Owner role**: Governance steward. **Delivery window**: Iteration 1.
- **FR-003**: The coordinator prompt MUST clarify that "continue" from the user means "advance to the next single boundary stop, then halt and ask." Squad MUST treat "continue" as a single-step instruction, not a license for autonomous multi-boundary advance. **Owner role**: Governance steward. **Delivery window**: Iteration 1.
- **FR-004**: The coordinator prompt MUST include a worked example showing the compliant 7-authorization-per-iteration pattern with explicit authorization phrasing for each boundary (matching the dogfooding pattern established in Features 011-014). **Owner role**: Governance steward. **Delivery window**: Iteration 1.
- **FR-005**: The coordinator prompt MUST include a worked example showing a violating bundled-advance pattern (Squad emitting review + retro + closeout commits without intervening authorization) and the validator FAIL response. **Owner role**: Governance steward. **Delivery window**: Iteration 1.
- **FR-006**: `validate-governance.ps1` MUST be extended with hard-validator rule `validation-fail.bundled-boundary-advance` that: detects when 2 or more boundary commits exist in the commit history since the most recent human authorization recorded in `.squad/decisions.md`, emits structured FAIL output naming the offending commit pair, the iteration, and the missing intervening authorization, and returns non-zero exit code so the validator can be used as a CI gate. **Owner role**: Validator steward. **Delivery window**: Iteration 1.
- **FR-007**: The validator MUST recognize the canonical boundary-commit signature patterns (subject-line regex). Recognition MUST be subject-line-pattern-based, not file-content-based, to keep the rule mechanical and fast. The canonical patterns are listed in the Implementation Boundary section. **Owner role**: Validator steward. **Delivery window**: Iteration 1.
- **FR-008**: Squad MUST auto-generate the canonical human-authorization shape in `.squad/decisions.md` from the user's authorization paste. Required captured fields: Decision ID, Type (`authorization` or `sign-off`), Boundary, Approving Human, Recorded At (ISO 8601 UTC), Commit Reference, Authorization Text (verbatim). The boundary inspection flow MUST allow the user to review or override the generated metadata before advancing. **Owner role**: Governance steward. **Delivery window**: Iteration 1.
- **FR-009**: Paired authorizations (hardening-gate sign-off + implementation authorization) MUST be recorded as TWO distinct entries in `.squad/decisions.md`, even when the user supplies one authorization paste. Squad MUST auto-generate the pair as separate entries; single-entry multi-boundary authorizations are rejected as bundled. **Owner role**: Governance steward. **Delivery window**: Iteration 1.

#### Pillar 2: Essence in Console

- **FR-010**: The coordinator prompt MUST require Squad's boundary handoffs to use Feature 014's three-section format AND populate each section substantively. For planning, implementation, review, and retro, "What I just did" MUST include at least 3 specific identifiers (commit hash, `file:///` path, FR-###, T###, or decision reference) AND at least 50 words. For iteration-closeout and feature-closeout, the section MAY satisfy a soft OR threshold (either the identifier count OR the word-count threshold). These thresholds are fixed for Feature 016 and MUST NOT be made tunable per project. "Why I stopped" MUST name the specific boundary phase being entered (not generic "next step"). "What I need from you" MUST be a specific actionable request naming the boundary, the inspection target(s), and the verdict required. This requirement applies to Squad's console handoffs only, not downstream artifact bodies. **Owner role**: Governance steward. **Delivery window**: Iteration 1.
- **FR-011**: `validate-governance.ps1` MUST be extended with soft-validator rule `soft-warning.thin-what-i-just-did` that fires when the "What I just did" section of a boundary handoff fails the fixed threshold for its lifecycle phase: planning, implementation, review, and retro require both the identifier count and the 50-word minimum; iteration-closeout and feature-closeout require at least one of those two thresholds. This rule remains soft-warning throughout Feature 016. **Owner role**: Validator steward. **Delivery window**: Iteration 1.
- **FR-012**: `validate-governance.ps1` MUST be extended with soft-validator rule `soft-warning.unspecific-stop-boundary` that fires when "Why I stopped" content does not name the specific boundary phase. The validator MUST cross-reference the current iteration state to detect mismatches (e.g., Squad says "next is review" but the iteration is at retro boundary). **Owner role**: Validator steward. **Delivery window**: Iteration 1.
- **FR-013**: `validate-governance.ps1` MUST be extended with soft-validator rule `soft-warning.unactionable-user-request` that fires once per handoff when "What I need from you" content does not name (a) the specific boundary being authorized, (b) the inspection target(s) as `file:///` references, AND (c) the verdict required. The emitted warning MUST list every missing component (`boundary-name`, `inspection-target`, `verdict-required`) in that single warning. **Owner role**: Validator steward. **Delivery window**: Iteration 1.
- **FR-014**: The coordinator prompt MUST include worked examples of substantive vs thin handoffs for each lifecycle phase, with annotations explaining what makes each section substantive. **Owner role**: Governance steward. **Delivery window**: Iteration 1.

#### Pillar 3: Click-Through Navigation

- **FR-015**: The coordinator prompt MUST require `file:///` URL format for ALL artifact references in Squad's narration and boundary stop messages. Bare relative paths or bare absolute paths in narration are forbidden. **Owner role**: Governance steward. **Delivery window**: Iteration 1.
- **FR-016**: `validate-governance.ps1` MUST implement the bare-path-in-boundary-handoff rule as a parameterized severity rule so rollout can change by configuration rather than rewrite. In Iteration 1, the rule emits `soft-warning.bare-path-in-boundary-handoff`; in Iteration 2, after exemption-list integration tests prove bounded false positives, the same rule flips to `validation-fail.bare-path-in-boundary-handoff`. The rule fires when a boundary stop message contains a bare path reference outside approved exemption contexts. **Owner role**: Validator steward. **Delivery window**: Iteration 1-2.
- **FR-017**: `validate-governance.ps1` MUST be extended with soft-validator rule `soft-warning.bare-path-in-narration` that fires when non-boundary Squad text (in-flight progress updates, tool-call narration) contains bare path references. Lower severity than the boundary-handoff rule because in-flight narration is less critical for click-through. **Owner role**: Validator steward. **Delivery window**: Iteration 1.
- **FR-018**: The validator MUST define explicit exemptions for paths in: shell-command arguments (e.g., `git add specs/...`), inline-code blocks, log output, JSON/YAML literals, regex patterns, and file glob arguments. Projects MAY extend that exemption list via `.specrew/config.yml`, but each extension MUST carry recorded human approval with approver name and rationale so exemption growth remains reviewable. These contexts contain paths but are not intended for click-through navigation. **Owner role**: Validator steward. **Delivery window**: Iteration 1.
- **FR-019**: When Squad cites an artifact via `file:///` URL, the validator MUST check that the file actually exists at that path. Citations to non-existent paths emit soft-warning `soft-warning.broken-file-url-reference`. **Owner role**: Validator steward. **Delivery window**: Iteration 1.

#### Cross-Cutting Requirements

### Key Entities *(include if feature involves data)*

- **Boundary Authorization Record**: A `.squad/decisions.md` entry auto-generated by Squad from a user authorization paste, capturing the approving human, the boundary being authorized, the ISO 8601 UTC authorization time, the commit reference, and the verbatim authorization text, with user review/override available during boundary inspection.
- **Boundary Handoff Message**: The three-section Squad-authored console stop message that summarizes completed work, identifies the exact boundary, and asks for a specific next decision.
- **Boundary Commit Signature**: The canonical subject-line pattern that allows the validator to classify lifecycle boundary commits mechanically.
- **Interaction-Model Validator Rule**: A hard-fail or soft-warning rule that enforces one of the three pillars or a related navigation/substance check against Squad-authored outputs only.
- **Known Trap Row**: A corpus entry that documents a recurring governance failure pattern, its enforcement status, and its linked tests and requirements.

## Non-Functional Requirements

- **NFR-004**: The feature MUST be backward compatible. Pre-Feature-016 iterations MUST continue to validate (grandfathered, consistent with Feature 013's pattern).
- **NFR-005**: The soft-warning emissions MUST remain advisory. Squad's responses MUST NOT be blocked by these warnings; the validator MUST continue to exit with success status when only soft warnings are present (hard-fail rules block; soft warnings inform). The Pillar 2 soft-warning rules remain advisory for the entirety of Feature 016.
- **NFR-006**: All emission output MUST be human-readable. Each warning/FAIL MUST name the specific phrase or pattern triggering it, the iteration/feature in scope, and a remediation hint (e.g., "add file:///prefix to the bare path /Dev/Specrew/specs/...").

## Risks and Mitigations

**Risk 1**: The coordinator prompt update destabilizes existing dogfooding patterns.  
**Mitigation**: Dogfooding already follows the 7-authorization discipline implicitly via user habit. The new prompt formalizes what is already happening. Backward compatibility is preserved.

**Risk 2**: The hard-validator rules block legitimate workflow when commits land for legitimate reasons without `.squad/decisions.md` entries or when handoff messages have ambiguous content.  
**Mitigation**: NFR-004 grandfathers pre-Feature-016 iterations. For post-feature work, the validator failure surfaces the missing piece with a clear remediation message.

**Risk 3**: External-project users find the new discipline restrictive.  
**Mitigation**: This IS Specrew's value-add over vibe coding. The restriction is the product. The README and handoff template explain why each pillar matters.

**Risk 4**: The validator misclassifies "substantive" content — judgment is inherently fuzzy.  
**Mitigation**: Use mechanical rules (identifier count, word count) rather than semantic analysis. False positives are recoverable (user can add an identifier); false negatives are caught in the next dogfooding cycle and become test fixtures.

**Risk 5**: The exemption list for bare paths is incomplete and produces false positives in legitimate contexts.  
**Mitigation**: FR-018 explicitly enumerates the exemptions, allows project-specific extension only with recorded human approval, and FR-021 requires exemption-list integration tests before the severity flip. New exemption cases discovered in dogfooding become reviewed test fixtures and exemption-list additions.

**Risk 6**: The "essence in console" rule requires Squad to write substantive content, which adds latency and tokens.  
**Mitigation**: This is the explicit trade-off the user is requesting. The latency/token cost is the price of visibility. Squad's narration was already verbose; the new rules redirect verbosity from procedural narration to substantive content.

**Risk 7**: Three new soft-warning rules + two new hard-fail rules + multiple coordinator prompt updates is a lot of surface change. Risk of incomplete implementation or rule interaction bugs.  
**Mitigation**: The two iterations split the work cleanly — Iteration 1 ships rules, Iteration 2 ships tests + docs. The integration tests in Iteration 2 catch rule-interaction bugs before feature closeout.

**Risk 8**: Squad's substantive content might unintentionally include sensitive information that doesn't belong in a public artifact.  
**Mitigation**: The substantive content is summaries of decisions/changes already in the `.md` files. If the artifacts don't contain sensitive content, the summaries don't either. The existing security-surface concern in hardening gates already addresses this.

## Implementation Boundary

The canonical boundary-commit signatures the validator MUST recognize (subject-line regex). The first 7 rows are the per-iteration boundary set; `feature-closeout` is a separate feature-level boundary outside that per-iteration count:

| Boundary | Subject-line regex |
| --- | --- |
| Planning | `^Feature \d+.* iteration \d+ planning boundary` |
| Hardening-gate sign-off + implementation auth | `^Feature \d+.* iteration \d+: record hardening-gate sign-off and implementation authorization` |
| Implementation | `^Feature \d+.* iteration \d+: implement` OR `^Feature \d+.* iteration \d+: bounded` |
| Review boundary | `^Feature \d+.* iteration \d+ review boundary` |
| Review verdict sign-off | `^Feature \d+.* iteration \d+ review-verdict-signoff boundary` |
| Retro boundary | `^Feature \d+.* iteration \d+ retrospective boundary` |
| Iteration closeout | `^Feature \d+.* iteration \d+ closeout boundary` |
| Feature closeout | `^Feature \d+.*: feature-closeout boundary` |

The canonical human-authorization entry shape in `.squad/decisions.md`:

```markdown
## YYYY-MM-DDTHH:MM:SS+TZ — Authorization: <boundary-name>

- **Decision ID**: authorization-<feature-NNN>-iter-<NNN>-<boundary-name>
- **Type**: authorization | sign-off
- **Boundary**: <hardening-gate-signoff | implementation | review-boundary | review-verdict-signoff | retro-boundary | iteration-closeout | feature-closeout>
- **Approving Human**: <human name>
- **Recorded At**: <ISO 8601 UTC timestamp>
- **Commit Reference**: <pending | short-hash>
- **Authorization Text**: <verbatim authorization phrase from user>
```

Squad auto-generates these entries from the user's authorization block. When one pasted authorization covers both hardening-gate sign-off and implementation authorization, Squad MUST emit two distinct entries rather than a single combined record. The user MAY review or override generated metadata during boundary inspection before advancement.

The substantive-content thresholds (FR-010 / FR-011):

- "What I just did" MUST contain at least 3 specific identifiers drawn from commit hashes, `file:///` URLs, FR references, task identifiers, or authorization references.
- "What I just did" MUST be at least 50 words for planning, implementation, review, and retro.
- Planning, implementation, review, and retro use a strict AND threshold (identifier count AND word count).
- Iteration closeout and feature closeout use a soft OR threshold (identifier count OR word count).
- These thresholds are fixed for Feature 016 and are not tunable per project.

The bare-path exemption pattern list (FR-018):

- Within fenced code blocks
- Within inline code spans
- After shell-command verbs in plain text
- Within JSON or YAML literal values
- Within regex pattern syntax
- Within existing URL contexts
- Within project-specific exemption contexts explicitly approved in `.specrew/config.yml` with approver name and rationale

The bare-path-in-boundary-handoff rule MUST inspect Squad-authored handoffs only. It does not parse conversation transcripts or user-typed text. Its severity is parameterized: soft-warning in Iteration 1, then hard-fail in Iteration 2 after exemption-list integration tests prove bounded false positives.

The actionable-request rule emits one warning per handoff and names each missing component when `boundary-name`, `inspection-target`, or `verdict-required` is absent.
