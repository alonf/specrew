# Onboarding Text Surface Contract

**Feature**: `010-onboarding-resume-visibility`  
**Contract type**: Text surface content contract  
**Version**: 1.0.0  
**Date**: 2026-05-10  
**Status**: Approved for implementation

---

## Purpose

This contract specifies the **minimum required content** that each onboarding surface must present after this feature is implemented. It is the authoritative reference for implementation and review. Any implementation that satisfies every clause in this contract passes the content requirement for this feature.

This contract does not prescribe exact wording. It specifies required semantic elements, required ordering constraints, and required omissions. Implementers have discretion over phrasing provided all semantic obligations are met.

---

## Surface 1 — `README.md`

**FR traceability**: FR-001, FR-004  
**Delivery type**: Primary delivery target

### Required Elements

| Clause | Semantic Obligation | Placement |
|--------|---------------------|-----------|
| `README-C1` | MUST state that resumed sessions (sessions after the first) also begin with `specrew start` | Recommended flow section or Notes section |
| `README-C2` | MUST name `copilot` explicitly as the unsupported launch path | Co-located with or immediately adjacent to README-C1 |
| `README-C3` | MUST include a one-sentence rationale explaining why running `copilot` directly is unsupported (i.e., it bypasses runtime handoff regeneration or equivalent brief explanation) | Same location as README-C2 |

### Forbidden Content

- MUST NOT imply `specrew start` is only needed on the first session.
- MUST NOT present running `copilot` directly as a valid or acceptable Specrew launch path.

### Visibility Constraint

The resume guidance and anti-pattern warning MUST be findable without reading the entire document. If placed in the Recommended flow, they may appear as a follow-on note to step 2. If placed in Notes, they must be the first or second note item.

### Example (non-normative)
>
> **Resuming work on a later day**: Run `specrew start` again — every session, including resumes, begins with `specrew start`. Running `copilot` directly is not supported: it skips the runtime handoff regeneration that `specrew start` performs before launch.

## Surface 2 — `docs/getting-started.md`

**FR traceability**: FR-002, FR-004  
**Delivery type**: Primary delivery target

### Required Elements

| Clause | Semantic Obligation | Placement |
|--------|---------------------|-----------|
| `GS-C1` | MUST include a "Resuming work later" subsection heading (or equivalent prominent heading) | After the Greenfield Quickstart first-session guidance, before Brownfield or at end of Greenfield |
| `GS-C2` | Subsection MUST name `specrew start` as the command for every resumed session | Inside the subsection |
| `GS-C3` | Subsection MUST explicitly list the three transient runtime files regenerated by `specrew start`: `.specrew/last-start-prompt.md`, `.specrew/start-context.json`, `.specrew/start-summary.md` | Inside the subsection |
| `GS-C4` | Subsection MUST explain that these transient files do not travel with git | Inside the subsection |
| `GS-C5` | Subsection MUST state that `specrew start` rebuilds them from tracked iteration state on each machine (cross-machine case) | Inside the subsection |
| `GS-C6` | Subsection MUST name `copilot` explicitly as the unsupported launch path | Inside the subsection |
| `GS-C7` | Anti-pattern warning MUST include a one-sentence rationale | Inside the subsection |
| `GS-C8` | Subsection SHOULD clarify that resuming means starting a new session, not re-running `specrew start` inside an already-active conversation | Inside the subsection (edge case from spec) |

### Forbidden Content

- MUST NOT imply `specrew start` is a first-launch-only step anywhere in the document.
- MUST NOT present running `copilot` directly as a valid or acceptable Specrew launch path.

### Visibility Constraint

The "Resuming work later" subsection MUST appear as a clearly delineated section findable through standard document navigation (heading in the Markdown table of contents, if present, or visible without full-document read). It must not be buried in an Appendix or Troubleshooting section.

### Platform Neutrality

All wording must apply equally to Windows, macOS, and Linux. No Windows-only paths or commands in the resume guidance.

### Example subsection structure (non-normative)

```markdown
### Resuming work later

Every subsequent session — whether you are on the same machine or a different one —
begins with `specrew start`, not with `copilot` directly.

```powershell
pwsh -File C:\Dev\Specrew\scripts\specrew.ps1 start
```

`specrew start` regenerates the per-machine runtime handoff files before launching Copilot:

- `.specrew/last-start-prompt.md`
- `.specrew/start-context.json`
- `.specrew/start-summary.md`

These files are transient: they are created for the current machine from the tracked iteration state
in the repository and are not committed to git. Running `copilot` directly skips this regeneration
step and is not a supported Specrew launch path.

**Cross-machine resumes**: After pulling the latest repository state on a new machine, run
`specrew start` as usual. It will rebuild the runtime handoff locally from the tracked project state.

**Active sessions**: If you are already inside a live Squad session launched by `specrew start`,
do not re-run `specrew start` in the same conversation. Resuming refers to starting a new session
in a new terminal, not restarting within an active conversation.

```

## Surface 3 — Bootstrap Complete Banner (`scripts/specrew-init.ps1`)

**FR traceability**: FR-003, FR-004  
**Delivery type**: Primary delivery target; only allowed code-shaped change in this feature

### Required Elements

| Clause | Semantic Obligation | Placement |
|--------|---------------------|-----------|
| `BANNER-C1` | MUST include a "Resuming work later" line or item in the Next Steps block | Inside `Write-PostBootstrapGuidance`, within the `=== Next Steps ===` block |
| `BANNER-C2` | Line MUST state that every subsequent session runs through `specrew start` | Same item |
| `BANNER-C3` | Line MUST reference that `specrew start` regenerates the runtime handoff before launch | Same item (brief phrasing acceptable for terminal context) |
| `BANNER-C4` | MUST state that running `copilot` directly is not the supported path | Same item or adjacent item |
| `BANNER-C5` | Anti-pattern statement MUST include a one-sentence rationale | Same item or adjacent item |

### Implementation Constraint
Only `Write-Host` lines within the `Write-PostBootstrapGuidance` function body may be added or modified. No logic, parameters, or other functions in `scripts/specrew-init.ps1` may be changed.

### Visibility Constraint (SC-005)
The resume guidance MUST be visible without scrolling on a standard terminal width of at least 100 columns. Lines emitted by `Write-Host` should stay within 100 characters where possible. Multi-line emission is acceptable.

### Forbidden Content
- MUST NOT change any control flow, parameter handling, or function signature in `specrew-init.ps1`.
- MUST NOT add new functions or restructure the banner.

### Example addition to Next Steps block (non-normative)
```powershell
Write-Host '2. Resume later sessions:' -ForegroundColor Yellow
Write-Host '   - Every subsequent session also begins with specrew start — not with copilot directly.' -ForegroundColor White
Write-Host '   - specrew start regenerates the runtime handoff (.specrew/last-start-prompt.md and friends) before launch.' -ForegroundColor White
Write-Host '   - Running copilot directly bypasses that regeneration and is not a supported Specrew launch path.' -ForegroundColor White
Write-Host '' 
```
