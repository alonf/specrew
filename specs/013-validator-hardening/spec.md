## Problem Statement

Specrew's governance validator is the canonical mechanical check surface for feature and iteration artifacts, but recent dogfooding exposed six rigor gaps that still rely on human review or fail with raw script exceptions. The current process can miss canonical hardening-gate deviations, reused approval evidence, and lifecycle over-claims, and it can also crash when iteration state metadata uses non-canonical field names. This feature closes those gaps so validator failures fail closed with clear, actionable messages instead of silent acceptance or PowerShell exceptions.

## Relationship to Existing Features

- **Spec 005 mechanical checks**: This feature extends the deterministic, fail-closed validator layer and does not introduce model-based review.
- **Spec 007 user-facing-progress-handoff**: Any surfaced validator failure must follow clear human-readable error reporting rather than raw stack traces.
- **Spec 008 reviewer-escalation symmetry**: Stronger validator enforcement reduces the load on reviewer-only catches.
- **Specs 005, 008, 009, 010, 011, and 012 corpus learnings**: This feature graduates documented validator traps from passive guidance into enforced rules.

## Clarifications

### Session 2026-05-12

- Q: What is the canonical iteration `state.md` schema? → A: Eight fields in canonical `**Field Name**:` format — Schema, Last Completed Task, Tasks Remaining, In Progress, Baseline Ref, Updated, Current Phase, Iteration Status. Additional narrative sections may exist but cannot replace these fields.
- Q: What values are allowed for `Iteration Status`? → A: Free-form descriptive text is allowed. The validator only inspects the value for closure-oriented keywords such as `closed` or `closeout complete`.
- Q: Where do the canonical contracts live? → A: Under `specs/013-validator-hardening/contracts/iteration-state-schema.md` and `specs/013-validator-hardening/contracts/hardening-gate-concerns.md`, matching the repository's existing feature-local contract pattern (for example `specs/005-stack-aware-quality-bar/contracts/quality-governance-artifacts.md`).
- Q: How should pending pre-implementation values be handled? → A: Pending values are valid so long as the canonical field names are present.
- Q: What counts as duplicated approval evidence? → A: Quotes that match after whitespace normalization and markdown-emphasis stripping, unless the artifact explicitly declares a blanket multi-iteration authorization scope.
- Q: Which `.github/copilot-instructions.md` changes are bookkeeping only? → A: Timestamp, `## Active Technologies`, and `## Recent Changes` updates generated by `/speckit.plan`; changes outside those areas are behavior-affecting.
- Q: Where should the `.github/copilot-instructions.md` bookkeeping classifier live? → A: As a standalone helper consumed by `scripts/specrew-start.ps1`; `validate-governance.ps1` may reuse or validate its result, but restart-policy ownership must stay outside the validator entrypoint.
- Q: What file scope counts for the FR-004 working-tree-differences over-claim check? → A: Dirty-tree enforcement is limited to the iteration directory's canonical artifacts. Repo-level governance traces such as `.squad/decisions.md` and `.squad/identity/now.md` may still be consulted as evidence sources, but they do not fail the dirty-tree check by themselves.
- Q: How are pre-existing iterations treated under canonical-schema enforcement? → A: Iterations that already existed before feature 013 rollout are grandfathered rather than retrofitted solely for FR-001 compliance; the canonical schema is required for new iterations and for older iterations only when they are materially reopened or rewritten.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Enforce canonical iteration metadata (Priority: P1)

A human developer or planner author who writes a non-canonical iteration `state.md` receives a clear validator failure naming the file, the missing canonical field, and a remediation hint instead of a PowerShell crash.

**Why this priority**: Validator crashes block the governance lane and hide the real problem behind tooling failure.

**Independent Test**: Create an iteration `state.md` with non-canonical field names or missing canonical fields, run the validator, and confirm it emits structured FAIL output with remediation guidance and no unhandled exception.

**Acceptance Scenarios**:

1. **Given** an iteration `state.md` uses non-canonical field names such as `Overall Status` instead of `**Iteration Status**:`, **When** the validator runs, **Then** it emits a structured FAIL naming the file, the missing canonical field, and a remediation hint without throwing an unhandled exception.
2. **Given** an iteration `state.md` contains all eight canonical fields in canonical format, **When** the validator runs, **Then** the iteration passes the schema check.
3. **Given** an iteration `state.md` is missing one or more canonical fields, **When** the validator runs, **Then** the FAIL output lists each missing field.

---

### User Story 2 - Enforce canonical hardening-gate concerns (Priority: P1)

A human developer or planner author who writes a `quality/hardening-gate.md` that drops or reorders the five canonical concerns receives a clear validator failure naming the missing or out-of-order concern.

**Why this priority**: The canonical five concerns are part of the hardening-gate contract and must not silently drift.

**Independent Test**: Create a `hardening-gate.md` that omits or reorders a canonical concern, run the validator, and confirm it reports the exact concern and expected position.

**Acceptance Scenarios**:

1. **Given** a `hardening-gate.md` Concern Review table is missing one canonical concern, **When** the validator runs, **Then** it emits a FAIL naming the missing concern and required row position.
2. **Given** a `hardening-gate.md` Concern Review table includes the canonical concerns out of order, **When** the validator runs, **Then** it emits a FAIL naming the ordering problem.
3. **Given** a `hardening-gate.md` lists the canonical five concerns first and then adds feature-specific concerns, **When** the validator runs, **Then** it passes the canonical-concern check.

---

### User Story 3 - Detect reused approval evidence across iterations (Priority: P1)

A human developer or planner author who reuses a prior iteration's approval evidence quote sees a validator failure naming both iterations and the duplicated quote unless the artifact explicitly declares a blanket multi-iteration authorization scope.

**Why this priority**: Approval-evidence reuse is a recurring governance trap that should be caught mechanically instead of by later human cleanup.

**Independent Test**: Create sibling iteration artifacts with byte-identical approval evidence quotes, run the validator, and confirm it reports the duplication while allowing explicit blanket-authorization cases.

**Acceptance Scenarios**:

1. **Given** two sibling iterations contain matching approval evidence quotes, **When** the validator runs, **Then** it emits a FAIL naming both iterations and the duplicated quote.
2. **Given** a blanket multi-iteration authorization is explicitly labeled with its covered scope, **When** the validator runs, **Then** the cited iterations pass the reuse check.
3. **Given** an iteration cites reused approval evidence without explicit blanket-scope labeling, **When** the validator runs, **Then** it emits a FAIL.

---

### User Story 4 - Block unsupported closeout claims (Priority: P1)

A human reviewing an iteration that claims closure sees validator failures when the required closeout evidence is missing, inconsistent, or still uncommitted.

**Why this priority**: Over-claim patterns have recurred across multiple features and undermine trust in lifecycle status.

**Independent Test**: Mark an iteration as closed while removing required closeout artifacts or leaving iteration-directory changes uncommitted, then run the validator and confirm it reports each missing or inconsistent evidence item.

**Acceptance Scenarios**:

1. **Given** an iteration claims a closed status but is missing `retro.md`, **When** the validator runs, **Then** it emits a FAIL naming the missing retro.
2. **Given** an iteration claims a closed status but `review.md` is missing or not accepted, **When** the validator runs, **Then** it emits a FAIL naming the review problem.
3. **Given** an iteration claims a closed status but `quality/hardening-gate.md` lacks post-implementation evidence for required concerns, **When** the validator runs, **Then** it emits a FAIL naming the pending evidence.
4. **Given** an iteration claims a closed status but the iteration directory still has uncommitted changes, **When** the validator runs, **Then** it emits a FAIL naming the changed files.

---

### User Story 5 - Improve validator failure reporting and restart guidance (Priority: P2)

A human running the validator receives structured, actionable FAIL output for every failure mode, and bookkeeping-only regenerations of `.github/copilot-instructions.md` are distinguished from behavior-affecting changes so unnecessary restarts are avoided.

**Why this priority**: Clear failures and fewer needless restarts reduce repeated development friction across every feature boundary.

**Independent Test**: Exercise parse failures, missing files, schema deviations, and `.github/copilot-instructions.md` diff classifications and confirm that all user-visible failures are structured and that bookkeeping-only changes are not treated as restart triggers.

**Acceptance Scenarios**:

1. **Given** any parse failure, missing required file, or schema deviation, **When** the validator encounters it, **Then** it emits a structured FAIL with file path, line number where applicable, category, and remediation hint, exits non-zero, and does not raise an unhandled exception.
2. **Given** `.github/copilot-instructions.md` changes only in timestamp, `Active Technologies`, or `Recent Changes`, **When** the classifier evaluates the diff, **Then** it classifies the change as bookkeeping and does not mark it as a session-restart trigger.
3. **Given** `.github/copilot-instructions.md` changes any behavior-affecting section, **When** the classifier evaluates the diff, **Then** it classifies the change as behavior and marks it as a restart trigger.

---

### Edge Cases

- Pre-implementation iterations may contain pending values in canonical fields; those values are valid if the canonical field names are present.
- Iteration `state.md` files may include additional narrative sections; the validator must enforce the canonical fields without rejecting extra explanatory content.
- Over-claim checks should focus on the iteration directory's canonical artifact set so unrelated trace files outside that scope do not create false positives; `.squad/decisions.md` and `.squad/identity/now.md` remain evidence sources, not dirty-tree blockers.
- Explicit blanket multi-iteration authorization labels make approval-evidence reuse acceptable for the covered iterations.
- If `.github/copilot-instructions.md` mixes bookkeeping-only edits with behavior edits in one change set, the more conservative behavior classification wins.
- Iterations that predate this feature's rollout are grandfathered for canonical-schema enforcement unless they are materially reopened or rewritten after rollout.
- Repositories with zero prior iterations must pass these rules gracefully without crashes.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST verify that every newly created or materially updated iteration `state.md` after rollout contains the canonical metadata fields `**Schema**`, `**Last Completed Task**`, `**Tasks Remaining**`, `**In Progress**`, `**Baseline Ref**`, `**Updated**`, `**Current Phase**`, and `**Iteration Status**` using the canonical `**Field Name**:` pattern, and MUST emit a structured FAIL for missing or non-canonical fields. Iterations that already existed before this feature's rollout are grandfathered unless they are materially reopened or rewritten. **Owner role**: Validator maintainers and governance-contract stewards. **Delivery window**: Iteration 1.
- **FR-002**: The system MUST verify that every `quality/hardening-gate.md` Concern Review table contains the five canonical concerns — `security-surface`, `error-handling-expectations`, `retry-idempotency-requirements`, `test-integrity-targets`, and `operational-resilience-concerns` — as the first five rows in that exact order, while allowing additional feature-specific concerns afterward. **Owner role**: Validator maintainers and hardening-gate contract stewards. **Delivery window**: Iteration 1.
- **FR-003**: The system MUST scan sibling iterations within a feature for duplicated Implementation Approval evidence quotes in `plan.md` and `state.md`, MUST flag reuse with a structured FAIL naming both iterations, and MUST allow reuse only when an explicit blanket-authorization scope is recorded. **Owner role**: Validator maintainers and governance-review stewards. **Delivery window**: Iteration 2.
- **FR-004**: The system MUST verify that any iteration whose `Iteration Status` indicates closure has the full closeout evidence set: accepted review, retro, post-implementation hardening evidence for required concerns, and no uncommitted changes inside the iteration directory. The dirty-tree check MUST scope to files inside the iteration directory's canonical artifact set; adjacent repo-level governance traces such as `.squad/decisions.md` and `.squad/identity/now.md` may provide evidence but MUST NOT fail the dirty-tree check by themselves. Missing or inconsistent evidence MUST emit a structured FAIL. **Owner role**: Validator maintainers and lifecycle-governance stewards. **Delivery window**: Iteration 2.
- **FR-005**: The system MUST convert parse failures, missing files, schema deviations, and unexpected validator conditions into structured FAIL output that includes file path or `(none)`, line number when applicable, error category, and remediation hint, and MUST complete the validation pass without exposing raw PowerShell exceptions to the user. **Owner role**: Validator maintainers. **Delivery window**: Iteration 1.
- **FR-006**: The system MUST classify changes to `.github/copilot-instructions.md` as `bookkeeping` when they are limited to timestamp, `Active Technologies`, or `Recent Changes`, and as `behavior` when any other section changes; bookkeeping changes MUST NOT require session restart guidance, while behavior changes MUST. The classifier MUST be implemented as a reusable helper consumed by `scripts/specrew-start.ps1` rather than embedded only inside `validate-governance.ps1`. **Owner role**: Validator maintainers and restart-policy stewards. **Delivery window**: Iteration 2.
- **FR-007**: The system MUST update `.specrew/quality/known-traps.md` so corpus rows for per-iteration approval evidence reuse, canonical concern enumeration, over-claim, and canonical iteration schema are marked as validator-enforced with citations to the implementing requirements and proving tests. **Owner role**: Governance-corpus stewards. **Delivery window**: Iteration 2.
- **FR-008**: The system MUST provide deterministic integration coverage for each new validator rule in this feature using violating and compliant fixtures so the validator's fail-closed behavior is proven through scaffold-replay-style assertions rather than ad hoc runtime checks. **Owner role**: Test maintainers and validator maintainers. **Delivery window**: Iterations 1 and 2.
- **FR-009**: The system MUST maintain a canonical contract for the iteration `state.md` schema and the canonical hardening-gate concerns in repository-tracked feature contracts at `specs/013-validator-hardening/contracts/iteration-state-schema.md` and `specs/013-validator-hardening/contracts/hardening-gate-concerns.md`, unless a later repository-level promotion preserves the same normative content and traceable discoverability. **Owner role**: Governance-contract stewards. **Delivery window**: Iteration 1.
- **FR-010**: The system MUST preserve the existing validator command surface, argument expectations, exit-code behavior, and PASS/FAIL compatibility so this hardening remains additive to current workflows. **Owner role**: Validator maintainers. **Delivery window**: Iterations 1 and 2.

### Key Entities *(include if feature involves data)*

- **Canonical Iteration State Schema**: The eight required metadata fields in canonical `**Field Name**:` form.
- **Canonical Hardening-Gate Concerns**: The five required Concern Review rows that must appear first and in canonical order.
- **Blanket Authorization Scope**: An explicit label that states which iterations a shared approval statement covers.
- **Over-Claim**: A closed-status iteration that lacks one or more required closeout evidence elements.
- **Bookkeeping Change**: A `.github/copilot-instructions.md` change limited to timestamp, `Active Technologies`, or `Recent Changes`.
- **Behavior Change**: A `.github/copilot-instructions.md` change that touches any other section and therefore affects restart guidance.
- **Copilot Instructions Change Classifier**: A reusable helper that classifies `.github/copilot-instructions.md` diffs for `specrew-start.ps1` and any validator-facing checks.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of non-grandfathered iteration `state.md` files examined after rollout either conform to the canonical schema or produce explicit FAIL output naming each schema violation, with zero raw PowerShell exceptions reaching users.
- **SC-002**: 100% of `quality/hardening-gate.md` files examined after rollout either preserve the canonical five concerns in the required first-five order or produce explicit FAIL output naming the deviation.
- **SC-003**: 100% of seeded sibling-iteration fixtures with duplicated approval evidence are rejected unless they include an explicit blanket-authorization scope label.
- **SC-004**: 100% of seeded over-claim fixtures that lack required closeout evidence or contain uncommitted iteration-directory changes are rejected by the validator.
- **SC-005**: 0 unhandled validator exceptions occur across representative canonical, non-canonical, missing-file, malformed, and empty-value fixtures.
- **SC-006**: 100% of representative `.github/copilot-instructions.md` diff fixtures are correctly classified as bookkeeping or behavior.
- **SC-007**: The relevant known-traps corpus rows are marked as enforced with citations to their implementing requirements and proving tests before the feature is considered complete.

## Assumptions

- The governance validator remains the canonical mechanical enforcement surface for feature and iteration artifact quality.
- Repository-managed contracts and known-traps documentation remain the authoritative reference points for validator rules and their graduation status.
- Feature-local `contracts/` directories are an acceptable canonical contract location when the extension root does not already provide a matching normative contract surface.
- Git-based working-tree inspection is available when validating closeout evidence.
- Existing scaffold-replay-style validation tests can be extended for this feature's new rules.
- This feature is limited to validator hardening and restart-guidance classification; it does not introduce new planning, review, or implementation workflows.

## Non-Goals

- Adding model-based or lens-execution review to the validator.
- Changing the validator's established command-line surface or exit-code conventions.
- Rewriting historical closed iterations solely to satisfy the new rules, including already-landed iterations in features 005, 007, 008, 009, 010, 011, and 012.
- Implementing downstream restart handling inside other orchestration commands rather than providing the classifier needed by them.
- Creating new corpus rules beyond the documented rows this feature graduates.
