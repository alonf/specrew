# Feature Specification: Specrew — Spec-Governed AI Crew Operating Model

**Feature Branch**: `001-specrew-product`
**Created**: 2026-04-17
**Status**: Active 0.14.0
**Implementation Note**: 14 implementing features have shipped as of 2026-05-13, so this product vision is now active rather than draft-only.
**Input**: User description: "Build a project called Specrew — a spec-governed operating model for AI crews combining Spec Kit as the specification/governance layer with Squad as the persistent multi-agent runtime layer."

## Problem Statement

AI agent teams today suffer from a governance gap. Tools like Squad provide multi-agent orchestration — routing work, spawning specialists, managing memory — but they have no built-in concept of a specification as the authoritative source of truth. Tools like Spec Kit provide structured specification authoring and governance workflows, but they have no way to drive a persistent agent crew from the artifacts they produce.

The result is that:

- Agent teams drift from what was specified, making undocumented decisions that silently replace the intended design.
- There is no reliable way to plan iterations, measure effort, or run structured delivery cycles with AI crews.
- Human stakeholders lose visibility into whether the crew is building what was agreed upon.
- There is no standard way to evaluate whether an AI crew's process and output actually conform to a specification.

No existing product bridges these two layers. Users are forced to manually translate between spec artifacts and agent instructions, losing traceability and governance along the way.

## Product Vision

Specrew lets users run an AI crew from the spec. It connects Spec Kit (the specification and governance layer) with Squad (the persistent multi-agent runtime layer) so that a single source of truth — the spec — governs what the crew builds, how it plans, how it executes, and how it is evaluated.

Specrew is built as a Spec Kit extension plus Squad-native configuration surfaces. The Spec Kit extension (`specrew-speckit`) governs spec lifecycle, governance artifacts, and drift detection. Squad integration uses Squad's native runtime surfaces: skills deployed to `.copilot/skills/specrew-*/`, ceremonies and directives registered in `.squad/` config files, and team roles merged into `.squad/team.md`. In v1, all Squad integration is Markdown-based (skills, ceremony definitions, directive references); SDK-defined team configuration (`squad.config.ts`) is deferred to post-MVP. Together they create a structured operating model where every iteration is planned from the spec, every task is traceable to a requirement, and every review checks alignment between implementation and intent.

Specrew currently targets GitHub Copilot because Squad currently supports that path. In v1, Copilot remains the execution runtime and default workhorse. When Copilot Agent HQ exposes additional selectable agents such as Claude or Codex, Specrew can optionally use them for review-role delegation, gated only by explicit user consent at bootstrap. Specrew is still architected for future compatibility if Squad later supports additional first-class coding-agent runtimes.

Existing specs, governance, and team config are preserved — Specrew merges its baseline roles rather than overwriting.

Manual board management is fallback-only if automation fails. If Squad cannot actually populate/manage the board, a capability gap or blocker MUST be recorded rather than silently downgrading to manual management.

Users work with Spec Kit slash commands for specification authoring (`/speckit.specify`, `/speckit.plan`, `/speckit.tasks`), then Squad ceremonies for iteration execution (planning, review/demo, retrospective). Specrew hooks fire automatically during Spec Kit workflows to enforce governance. No wrapper commands needed.

### Session 2026-05-04

- Q: What is the downstream interaction model after bootstrap? → A: Guided Specrew wrapper. After `specrew init`, downstream users SHOULD start work with `specrew start`, not by manually launching Copilot and invoking Spec Kit commands one-by-one. `specrew start` is the canonical entrypoint for feature delivery: it launches or hands off to Squad, optionally passes a short plain-language feature request, tells Squad to drive the full Spec Kit lifecycle (`specify`, `clarify` when needed, `plan`, `tasks`, `implement`), and keeps the human developer in a question-answer role only when Squad lacks enough information to proceed safely. Squad MUST inspect current artifacts first, continue any in-progress feature automatically, and if no active work exists or the previous feature is complete, ask the human whether they want a fix or a new feature and gather the missing details through the normal Spec Kit-guided intake. The generated handoff context MUST record the approval mode and launch mode so manual continuation matches the intended launch contract. Because the latest Copilot CLI docs still describe an initial folder-trust step, Specrew MAY reduce but cannot guarantee elimination of every first-launch prompt.

When Claude and Codex are enabled for delegated use, Specrew MUST actually route eligible lifecycle work to them according to role policy or explicitly report why it fell back. When the same artifact fails a governance gate repeatedly, Specrew MUST escalate repair quality by increasing the reasoning tier, then reassigning the artifact to an independent owner, and de-escalating after the gate passes so the system does not burn tokens in endless low-reasoning repair loops.
- Q: How should `specrew start` handle a new brownfield project that has existing code, docs, and git history but no active Specrew feature yet? → A: In `brownfield-new` state, Specrew MUST not jump straight to asking broad feature questions. It MUST first analyze the existing repository evidence — including code structure, key manifests, Markdown/docs files, and recent git history — to reconstruct the current product/system baseline, infer likely technologies and domain concerns, and carry that discovery into the Squad handoff. Squad MUST then use that evidence to seed the starting spec context, negotiate any missing change intent through targeted interactive questions, and continue through the same clarify, team-shaping, team-presentation, and explicit implementation-start gates as greenfield work. Squad SHOULD propose concrete supplemental specialists tied to the detected stack/domain when the current roster is obviously missing needed expertise, asking the human only for approval/corrections and unresolved change intent.
- Q: How should Specrew treat Spec Kit clarification in the downstream lifecycle? → A: Specrew MUST treat clarification as an explicit gate between `speckit.specify` and `speckit.plan`, not as an invisible optional shortcut. After `speckit.specify`, Squad MUST either run `speckit.clarify` or record a concrete skip rationale before planning. For `new-feature` and `brownfield-new` starts, the default is to run `speckit.clarify`; skipping is allowed only when the generated spec is already materially complete for planning and the reason is recorded in the run log/decisions. When clarification runs, the resulting answers MUST be written back into the spec before planning begins.
- Q: How should `specrew update` behave by default, and what visibility should it provide for Specrew, Squad, and Spec Kit versions? → A: Specrew MUST provide a standalone `specrew update` command. Bare `specrew update` updates Specrew only and also notifies the user if newer Squad and/or Spec Kit versions are available. `specrew update --info` reports the current and latest known versions for Specrew, Squad, and Spec Kit without mutating the project. Users may opt into broader upgrades with explicit flags such as `--all` or platform-specific selectors.
- Q: What additional greenfield flow gaps from the latest ClipBoard3 run must become explicit product requirements? → A: For `greenfield-new` starts with no supplied feature request, Squad MUST actively initiate intake with an explicit interactive question instead of merely printing advisory text, and it MUST stay in intake/clarification mode until it has enough grounded scope to begin `speckit.specify`. After the spec and clarify pass are complete, Squad MUST analyze the resulting requirements and technology assumptions, propose any missing best-fit supplemental team members, present the full resulting team to the human developer including existing Specrew baseline roles and any user-added members, and ask the human developer to explicitly start implementation before entering `speckit.implement`.
- Q: How should Specrew express production-grade quality expectations without forcing every downstream project into the same fixed quality profile? → A: Specrew MUST encode quality governance as requirement-driven expectations rather than a one-size-fits-all checklist. The planning and review flow MUST evaluate which quality attributes are materially required for the project or feature — such as clean code, SOLID-friendly decomposition, semantic correctness, robustness, retries, idempotency, error handling, logging, telemetry, security, operability, and concurrency safety — and MUST verify only the applicable ones. The system SHOULD discourage ceremonial sophistication by requiring protocol fields, resilience paths, and architectural mechanisms to have enforced runtime semantics or be removed/deferred explicitly.
- Q: What end-of-feature information should Specrew provide to prepare a human developer for code review? → A: When a feature implementation is complete, Squad MUST provide a structured implementation briefing for the human developer before closing the run. That briefing MUST summarize what was built, explain how the delivered behavior maps back to the requirements, describe the main happy path and any important alternative flows, list the packages/dependencies the implementation uses and call out which ones are new to the project, summarize the testing strategy, and provide a clearly labeled estimate of test coverage/confidence rather than implying exact coverage if it was not measured directly.

### Session 2026-05-06

- Q: How should Specrew increase delivery throughput when a single specialty becomes the bottleneck? → A: After `speckit.specify` and clarification establish the feature scope, Specrew MUST be able to analyze likely delivery concurrency and, when justified, propose role-paired capacity tiers for the same specialty instead of only adding brand-new specialties. The default paired terminology is **Junior / Senior** for the same specialty (for example Junior Frontend Developer + Senior Frontend Developer or Junior Backend Developer + Senior Backend Developer), but downstream projects MAY override those default labels in `role-assignments.yml` while preserving the same routing semantics. These are distinct named team members with different capability and routing profiles, not anonymous clones of the same agent identity. Junior roles are optimized for bounded, lower-risk, faster-execution work; Senior roles are optimized for ambiguous, cross-cutting, higher-risk, or reviewer-gated work. Specrew MUST only recommend such paired roles when the spec, plan shape, and dependency graph suggest meaningful safe parallelism, and the human developer MUST explicitly approve the resulting team before implementation begins.
- Q: How should Specrew stage its own automated testing strategy after the core evaluation harness exists? → A: In a later iteration, Specrew SHOULD add a three-lane validation strategy: (1) deterministic governance/integration tests as the primary PR gate, (2) artifact/contract tests that validate prompts, handoff context, lifecycle decisions, and structured traces without requiring live agent execution, and (3) a small live Copilot/Squad smoke lane run on a scheduled or pre-release basis rather than on every PR. Live agent runs MUST persist structured traces, and important real-world failures SHOULD be converted into replayable regression fixtures.
- Q: How should Specrew handle Copilot versus optional delegated agents at bootstrap, and how should it make delegated usage visible at runtime? → A: Copilot is the mandatory host runtime for Specrew's Squad integration in v1 and MUST NOT be presented as an optional consented delegated agent. `specrew init` SHOULD always treat Copilot as enabled when Copilot CLI is available, and only ask consent for optional delegated families such as Claude and Codex. To make optional delegated agents materially useful, default role preferences SHOULD route review- and problem-solving-heavy work toward them when available: Reviewer SHOULD prefer Claude, Spec Steward SHOULD prefer Codex (with fallback to Claude when Codex is unavailable), Planner SHOULD prefer Claude for planning/problem-solving work, and Implementer SHOULD remain on Copilot by default. Specrew MUST also require visible runtime evidence for delegated routing by recording the requested agent family, effective agent family, concrete model ID, and fallback reason whenever a delegated assignment is used or cannot be honored.
- Q: How should Specrew react when review finds gaps, ambiguous requirements, or other spec-alignment failures? → A: Specrew MUST operate with a **no-gap policy** for lifecycle-governed work. A known gap discovered during review, governance, or validation MUST NOT roll invisibly into the next iteration. The system MUST either: (1) fix the gap in the current iteration before closure, or (2) obtain explicit human approval to defer it and record that defer in the governing artifacts. Reviewer behavior MUST be critical and evidence-driven rather than success-biased: for every hardened governance or lifecycle requirement under review, Specrew SHOULD classify whether it is implemented, enforced, observable, and documented, and MUST surface a gap ledger when any of those dimensions are missing. If review uncovers an ambiguity, contradiction, or missing decision in the spec, Specrew MUST ask the human developer targeted clarification questions, update the spec accordingly, and repair any affected plan/task/governance artifacts before the iteration can close as aligned.
- Q: How should Specrew visibly help a human reviewer understand what the agents built without forcing the reviewer to read every diff and prose artifact first? → A: Specrew MUST generate a fixed set of reviewer-facing iteration artifacts for code-touching iterations — code map, dependency report, coverage/test evidence, and reviewer index — plus a conditional security surface when security is materially in scope. At iteration close, Specrew MUST also mirror the headline review signals to the terminal so a reviewer can triage the iteration before opening files.
- Q: Should these reviewer artifacts always be produced, or only when certain quality attributes apply? → A: Code map, dependency report, coverage/test evidence, and reviewer index are unconditional for code-touching iterations. Security surface is attribute-driven: generate it only when security is declared as materially required quality work in the plan or when a security-focused team member is active.
- Q: Should reviewer visibility include cost/billing telemetry for delegated agents? → A: No. Specrew MUST NOT report token, dollar, or billing context in v1. It MAY surface reasoning-tier escalation history as a qualitative operational signal, but never as a numeric cost claim.
- Q: Should Specrew implement its own vulnerability or SAST engine as part of reviewer visibility? → A: No. Specrew MAY invoke recognized existing toolchain scanners when present and embed their output with tool name and exit code, but it MUST NOT fabricate vulnerability intelligence or implement its own scanner/database.
- Q: How should the reviewer summary behave in CI, redirected output, or `--quiet` mode? → A: Interactive runs MUST print a structured Reviewer Summary block automatically at iteration close as the primary reviewer-entry surface. Non-interactive runs MUST emit a single stable machine-parseable digest line beginning with `schema=v1` plus the reviewer-index path and suppress the visual summary.
- Q: Should the existing implementation briefing be merged into the new reviewer artifacts? → A: No. The implementation briefing remains the agent-authored narrative summary. The reviewer artifacts are mechanical evidence surfaces. The reviewer index links to both without conflating them.
- Q: How should Specrew use diagrams to help reviewers understand structure and flow? → A: Specrew SHOULD prefer Mermaid-based diagrams in v1 because they are Markdown-native and easier to view locally in editors or browsers. Reviewer-facing diagram surfaces MUST use local files and local-open behavior rather than depending on third-party hosted renderers by default.
- Q: Should reviewer artifacts for a closed iteration be updated when later iterations change the implementation? → A: No. Reviewer artifacts inside `iterations/<NNN>/` are historical snapshots for that iteration and MUST remain immutable once the iteration closes. If Specrew provides a "current system view", it MUST live in a separate mutable artifact outside the closed iteration snapshot so reviewers can distinguish history from current state.
- Q: How should Specrew classify git-tracked versus local runtime files, and can a developer continue the same iteration from another machine? → A: Specrew MUST distinguish shared governance truth from per-machine transient runtime artifacts. Stable downstream bootstrap/governance files (for example `.specrew\config.yml`, `.specrew\iteration-config.yml`, `.specrew\role-assignments.yml`), shared Squad routing/decision surfaces, and iteration artifacts such as `state.md` remain tracked so a developer can continue the same iteration on another machine after commit/push. Ephemeral runtime handoff outputs (for example `.specrew\last-start-prompt.md`, `start-context.json`, `start-summary.md`, and dry-run reports) MUST be treated as transient. `.squad\agents\<role>\history.md` is a per-machine notebook rather than shared project truth; anything that should bind the team across machines MUST be promoted into `.squad\decisions.md` or the tracked iteration artifacts.
- Q: How should Specrew help users checkpoint work in git without silently auto-committing? → A: Specrew SHOULD offer an explicit user-approved commit at iteration boundaries rather than auto-committing silently. In particular, after presenting the closeout review for an iteration, and before making the first file change for a subsequent iteration when the prior iteration still has uncommitted tracked changes, Specrew SHOULD offer to create a commit. If the user declines, Specrew SHOULD remind them that cross-machine continuation only reflects the last committed checkpoint.

## Personas

### P1 — Solo Builder

A developer or indie hacker who wants to run an AI crew on their own project. They write a spec, bootstrap Specrew, and let the crew deliver iteratively while they act as stakeholder, reviewer, and Spec Steward. They value speed and low ceremony but still want to know the crew is building what was specified.

### P2 — Tech Lead

A technical leader setting up an AI-assisted development workflow for a team or project. They care about governance, traceability, and iteration discipline. They assign the Spec Steward role, configure capacity, and use review/demo/retro ceremonies to keep the crew aligned. They want confidence that the AI crew's output matches the agreed spec — especially when multiple humans are involved.

### P3 — Extension Author

A developer who builds on top of Specrew by creating additional Spec Kit or Squad extensions. They need clear extension boundaries, stable integration surfaces, and documentation of what Specrew owns vs. what it delegates. They must be able to add capabilities without colliding with Specrew's governance model.

### P4 — Project Stakeholder

A non-technical or semi-technical person who wants visibility into what the AI crew is doing. They review demo outputs, read iteration summaries, and approve or redirect scope. They never interact with the crew directly — they work through the spec and the Spec Steward.

### P5 — Time-Constrained Senior Reviewer

A senior engineer reviewing AI-produced work under time pressure. They want to know within 30 seconds whether the iteration looks safe, what changed materially, and where to dig deeper next. Specrew owes them a terminal-first Reviewer Summary for fast triage and a reviewer index that points to the right evidence in the right order.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Bootstrap and Start Specrew in a New Project (Priority: P1)

A user has an empty or existing repository. They run `specrew init`, which detects whether Spec Kit and Squad are already present, installs or validates compatible versions, then scaffolds the necessary governance artifacts (constitution placeholder, iteration config, role assignments) and configures the Squad team with the five protected baseline roles deterministically using Squad's non-interactive initialization mode (bypassing Squad's team-member casting interview). Copilot is treated as the mandatory host runtime rather than an optional delegated choice, while optional delegated agent families such as Claude and Codex can be consented for Specrew-managed routing. Bootstrap completes without blocking on team composition decisions and leaves the repository in a state recognizable by the Squad coordinator as a configured operation-ready team (not an unconfigured scaffold). Upon completion, `specrew init` outputs explicit next-step guidance directly in the terminal: the next command to run (`specrew start`), concise flow orientation, and team extension instructions. The developer does not need to leave the terminal or read getting-started documentation for baseline orientation. After bootstrap succeeds, users can add domain-specific team members via `specrew team add <member-name> --role <role> --charter <charter-text>`, which atomically creates all required Squad artifacts (`.squad/team.md` entry, `.squad/agents/<member>/charter.md`, `.squad/agents/<member>/history.md`) in a single operation. When the user is ready to build or continue work, they run `specrew start` with or without an initial feature request and answer only the questions Squad cannot resolve from context. If the repository already contains a Specrew-managed roster, Squad MUST detect it as an operational team and continue with that roster rather than recasting a generic Squad team. For brownfield projects with existing Spec Kit specs or Squad team config, Specrew merges its baseline into the existing setup without overwriting.

**Why this priority**: Without bootstrap, nothing else works. This is the entry point for all users and must work reliably before any other story delivers value.

**Independent Test**: Create a new empty repository, run `specrew init`, and verify that Spec Kit and Squad are installed at compatible versions, all expected artifacts exist, the Squad team is configured with the five baseline roles, and the Spec Steward role is assigned. Repeat with a brownfield repo that has existing Spec Kit config and verify merging preserves existing artifacts.

### User Story 9 — Shape the Team for Concurrency with Junior/Senior Role Pairs (Priority: P2)

After the feature spec and clarify pass are grounded, a user wants Specrew to decide not only which specialties are missing, but also whether the planned work is large or separable enough to justify more execution capacity inside the same specialty. Squad analyzes the feature scope, likely workstreams, dependency graph, risk profile, and current roster, then proposes a final working team that may include paired same-specialty roles such as Junior Frontend Developer + Senior Frontend Developer or Junior Backend Developer + Senior Backend Developer. The Junior/Senior pair is not a duplicate unnamed clone: each member has its own charter, routing profile, and expected task class. During execution, Squad routes bounded, lower-risk, well-scoped tasks to the Junior role and routes ambiguous, cross-cutting, high-risk, or reviewer-sensitive work to the Senior role, allowing safe parallel execution when work can be partitioned cleanly.

**Why this priority**: Specrew already shapes teams after the spec becomes clear, but it currently treats that decision mostly as “which specialties are missing.” Adding explicit concurrency-aware team sizing lets Specrew increase throughput without abandoning governance, while still keeping the human in control of the final team.

**Independent Test**: Start a feature whose clarified scope clearly contains multiple independent frontend or backend workstreams plus higher-risk integration work, then verify Squad proposes a Junior/Senior same-specialty pair with an explicit rationale, requires human approval, and routes lower-risk vs. higher-risk work according to the paired-role policy once implementation begins.

**Acceptance Scenarios**:

1. **Given** a clarified feature with only one narrow implementation stream in a specialty, **When** Squad performs team shaping, **Then** it does not propose a Junior/Senior pair for that specialty just to inflate the team.
2. **Given** a clarified feature with multiple independent workstreams inside one specialty and enough safe parallel work to justify extra capacity, **When** Squad performs team shaping, **Then** it may propose a Junior/Senior pair for that specialty and explain why the added capacity is warranted.
3. **Given** Squad proposes a Junior/Senior pair, **When** it presents the final team to the human developer, **Then** it distinguishes the paired roles as separate named members with different task profiles rather than as anonymous duplicate copies of the same role.
4. **Given** the human developer does not approve the proposed Junior/Senior expansion, **When** Squad reaches the implementation gate, **Then** it preserves the approved roster only and does not silently add the paired roles.
5. **Given** a Junior/Senior pair exists for a specialty, **When** Squad routes implementation work, **Then** bounded, lower-risk, well-scoped tasks go preferentially to the Junior role while ambiguous, high-risk, cross-cutting, or reviewer-sensitive work goes preferentially to the Senior role.
6. **Given** two same-specialty roles are available concurrently, **When** Squad launches parallel work, **Then** it must partition ownership clearly so both agents are not editing the same logical task slice without an explicit coordination plan.
7. **Given** a Junior role completes work on a shared or risky surface, **When** the change reaches the review or integration gate, **Then** Squad may escalate that work to the Senior role or an independent reviewer before final acceptance when governance policy requires it.

### Edge Cases

- What happens when the spec is empty or has no functional requirements when an iteration is started? Specrew should refuse to plan and prompt the user to complete the spec first.
- What happens when the Spec Steward role is unassigned? Specrew should block iteration start and require assignment.
- What happens when Squad is unavailable or not installed? Specrew's Spec Kit extension should still function for governance workflows, but should clearly inform the user that crew execution requires Squad.
- What happens when an iteration is abandoned mid-execution? Specrew should mark incomplete tasks as deferred, record the abandonment reason, and make them available for the next iteration.
- What happens when the crew crashes or a task fails unexpectedly mid-execution? Specrew persists task state to disk after each task completes, so the user can run a resume command to pick up from the last completed task without losing progress.
- What happens when multiple specs exist in a project? Specrew should scope iteration planning to a single spec at a time and require the user to select which spec to deliver against.
- What happens when a brownfield project has Spec Kit at an incompatible version? `specrew init` reports the version requirement, suggests the upgrade command, and does not proceed until the user upgrades.
- What happens when a brownfield project has a Squad team with custom roles that overlap with Specrew's baseline role names? `specrew init` detects the overlap, warns the user, and asks whether to merge (adopt existing role config) or rename (suffix the existing role).
- What happens when only Specrew has an update available? Bare `specrew update` should update Specrew only and report that Squad and Spec Kit are already current or unchanged.
- What happens when `specrew update --all` finds a newer version for one platform but not the others? Specrew should apply the requested update scope per platform and produce a per-platform result summary rather than treating the whole run as all-or-nothing unless a compatibility rule requires a stop.
- What happens when the user runs `specrew update --info` in an offline or partially disconnected environment? Specrew should report the current installed versions, clearly mark which latest-version checks could not be confirmed, and avoid mutating the project.
- What happens when Squad detects multiple independent workstreams but they all touch the same high-conflict surface? Specrew should avoid recommending Junior/Senior same-specialty parallelism for that specialty unless it can partition ownership cleanly.
- What happens when a Junior/Senior pair is proposed mainly for speed but the clarified feature is still highly ambiguous? Specrew should prefer a smaller Senior-led team until the ambiguity is resolved rather than creating noisy parallel execution.
- What happens when a Junior role repeatedly fails governance or integration gates on the same specialty slice? Specrew should escalate the affected slice to the Senior role or reduce same-specialty concurrency rather than persisting in low-quality parallel loops.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST provide a Spec Kit extension (built from the official Spec Kit extension starter template) and Squad-native configuration surfaces. Squad integration MUST use Squad's native runtime layout: skills deployed to `.copilot/skills/specrew-*/SKILL.md`, ceremonies registered in `.squad/ceremonies.md`, and directives referenced via agent charters in `.squad/agents/*/charter.md`. Specrew MUST NOT create a separate `extensions/specrew-squad/` package or rely on Squad's marketplace plugin system for bundled distribution. Specrew MUST NOT depend on third-party scaffolding tools for its runtime architecture.
- **FR-002**: System MUST provide a standalone `specrew init` CLI/script (not a Spec Kit extension command) that works before Spec Kit or Squad are installed. It MUST own the full dependency lifecycle: detect existing Spec Kit and Squad installations, install missing dependencies at pinned compatible versions (Spec Kit >= 0.8.4, Squad >= 0.9.1), run `specify init` and `squad init --non-interactive` to create `.specify/` and `.squad/` directories deterministically without blocking on Squad's team-member casting interview, validate version compatibility of existing installations, then install/configure the Specrew Spec Kit extension, deploy Squad skills to `.copilot/skills/specrew-*/`, register ceremonies in `.squad/ceremonies.md`, scaffold governance artifacts (downstream constitution placeholder, iteration configuration, role assignments), and configure the Squad team with five protected baseline roles: Spec Steward, Planner, Implementer, Reviewer, and Retro Facilitator. Bootstrap MUST complete without user interaction for team composition — Squad's casting interview is bypassed during `specrew init`. The bootstrapped downstream repository MUST be left in a state recognizable by the Squad coordinator as a configured, operation-ready team (not an unconfigured scaffold requiring fresh team creation). Upon successful completion, `specrew init` MUST output explicit next-step guidance directly in the terminal: (1) the next command(s) to run (e.g., starting spec authoring with Spec Kit workflows), (2) concise flow orientation (baseline crew → specify features → plan iteration → execute), and (3) references to team extension commands without requiring the developer to leave the terminal for baseline orientation or read separate getting-started documentation. After bootstrap, Specrew MUST provide a command-driven team management interface (via CLI output and/or getting-started documentation) that allows users to add, update, and remove domain-specific team members (e.g., Security Analyst, UX Designer, DBA) without manually editing multiple `.squad/` files. Baseline roles are protected (cannot be removed), but downstream projects can freely add supplemental team members through the command interface.
- **FR-003**: The spec MUST be the authoritative source of truth. Agent memory, team decisions, and implementation artifacts MUST NOT silently replace or contradict the spec without going through a tracked change process.
- **FR-004**: System MUST define and assign a Spec Steward role responsible for keeping spec, plan, tasks, decisions, and implementation aligned.
- **FR-005**: System MUST support an iteration lifecycle with four phases: planning, execution, review/demo, and retrospective. Planning and Review/Demo are Specrew-defined ceremonies. Retrospective leverages Squad's built-in Retrospective ceremony mechanism. Execution is a phase but not a ceremony — it is routed work dispatched to agents. The three ceremony phases use Squad's ceremony infrastructure; execution uses Squad's task routing and skill invocation.
- **FR-006**: During planning, the system MUST generate an iteration plan with tasks mapped to spec requirements, each task having an effort estimate, an assigned owner, and traceability to its source requirement.
- **FR-007**: System MUST support configurable effort measurement with a defined effort unit and iteration capacity limit. *(Specrew v1 default: story points. This is a Specrew design choice, not a platform-level concept.)*
- **FR-008**: During execution, the Spec Steward MUST actively monitor for drift after each task completes (intended implementation: a Squad post-task hook — exact hook shape to be validated against Squad's SDK surface during Iteration 0). If the post-task hook surface is unavailable, the canonical fallback is: the drift-check skill runs as an automated step within the Review/Demo ceremony, checking all tasks in batch. Drift is still caught within the same iteration. Surfaced drift MUST include a specific citation of the requirement and the deviation.
- **FR-009**: During review/demo, the system MUST evaluate each delivered task against its source requirement and produce a per-task verdict (pass / needs-work / blocked).
- **FR-010**: During retrospective, the system MUST capture estimation accuracy, drift events, process adherence, and improvement actions.
- **FR-011**: System MUST distinguish between Specrew's own project governance and the governance artifacts it generates for downstream projects. It MUST NOT use its own constitution as the default for downstream projects.
- **FR-012**: System MUST detect hard-stop collisions with other Spec Kit or Squad extensions and inform the user with a clear explanation of the conflict and resolution options. It MUST NOT silently override other extensions.
- **FR-013**: System MUST integrate exclusively through Spec Kit extension capabilities and Squad extension/config/SDK capabilities — not through Copilot-specific or VS Code-specific APIs.
- **FR-014**: System MUST be architecturally separable from GitHub Copilot so that if Squad later supports other coding-agent runtimes, Specrew can adapt without a rewrite.
- **FR-015**: System MUST include an evaluation harness that can bootstrap a fresh project, run multiple iterations against a reference spec, and produce a structured report covering process quality and outcome quality. The harness is staged: the process-quality scorer (artifact checks, ceremony adherence, drift detection verification) is delivered in Iteration 2; the outcome-quality scorer (requirement coverage, acceptance scenario pass rate) and full end-to-end harness are delivered in Iteration 3.
- **FR-016**: System MUST preserve user customizations (downstream constitution, iteration config, role assignments) across Specrew upgrades.
- **FR-017**: When an iteration plan exceeds configured capacity, the system MUST flag the overcommitment and suggest task deferral based on requirement priority.
- **FR-018**: Each completed task MUST record the requirement it traces to, the agent that executed it, the effort spent, and the review verdict.
- **FR-019**: The system MUST persist iteration task state to disk after each task completes and provide a resume command that continues execution from the last completed task after a failure or interruption.
- **FR-020**: On brownfield projects, `specrew init` MUST detect existing Spec Kit specs, governance artifacts, Squad team config, and installed extensions, and MUST merge Specrew's baseline roles and config into the existing setup without overwriting user data. When existing dependency versions are incompatible, it MUST report the conflict and suggest an upgrade path without proceeding.
- **FR-021**: When multiple Copilot-accessible delegated agents are available AND enabled by the user, Specrew MUST route the Reviewer role and Spec Steward role to an agent *different from the one used for implementation* whenever possible. Reviewer SHOULD prefer Claude by default, Spec Steward SHOULD prefer Codex by default (falling back to Claude when Codex is unavailable), Planner SHOULD prefer Claude by default for planning/problem-solving work, and Implementer SHOULD remain on Copilot by default. Agent preference is per-role and configurable in `role-assignments.yml` via a `preferred_agent` field. This independent-perspective principle improves review quality by ensuring independent oversight while keeping Copilot as the default workhorse.
- **FR-022**: `specrew init` MUST detect GitHub Copilot availability and which optional Copilot / Agent HQ selectable delegated agents (such as Claude and Codex) are currently accessible to the user. Copilot is the mandatory Specrew host runtime in v1 and MUST be treated as enabled whenever Copilot CLI is available; it MUST NOT be presented as an optional delegated-consent question. Optional delegated agents are explicit flag-driven opt-in only, and the resulting enablement state MUST be persisted in `iteration-config.yml` under an `agents:` section. `specrew init` SHOULD NOT ask an interactive delegated-agent bootstrap question. Specrew does not collect, display, or reason about billing/cost context; any cost implications are between the user and GitHub. When no agent-selection flags are supplied, default to `copilot: enabled` and all other detected delegated agents `enabled: false`. Flags: `--agents=claude`, `--agents=codex`, `--agents=claude,codex`, `--agents=all`, `--no-agents`.
- **FR-023**: System MUST provide command-driven team management commands (`specrew team add`, `specrew team update`, `specrew team remove`) that allow users to manage domain-specific team members without manually editing multiple `.squad/` files. The `add` command MUST atomically create: (1) a new row in `.squad/team.md` outside the Specrew-managed baseline block, (2) `.squad/agents/<member>/charter.md` with the provided role definition, and (3) `.squad/agents/<member>/history.md` as an empty initialized file. The `update` command MUST modify existing member charter or metadata. The `remove` command MUST delete all associated member artifacts. All operations MUST validate that baseline roles are not modified or removed. All operations MUST provide clear success/failure feedback and handle edge cases (duplicate names, missing members, file permission issues) gracefully.
- **FR-024**: System MUST provide a standalone `specrew start` command as the canonical downstream entrypoint after bootstrap. `specrew start` MUST: (1) validate that Specrew, Spec Kit, and Squad are already bootstrapped for the target project, (2) accept an optional short feature request and/or resume target, where any provided feature text is treated as a plain-language request rather than a full spec document, (3) launch or hand off to Squad using the Squad coordinator, (4) default to intake/resume behavior when invoked without a feature request by having Squad inspect active Spec Kit/Specrew artifacts and continue in-progress work when appropriate, (5) instruct Squad to ask the human whether they want a fix or a new feature when no active work exists or the last feature is complete, (6) instruct Squad to assess whether additional specialist team members are needed before a brand-new feature begins and only ask about team changes when the current roster appears insufficient or the human explicitly wants to adjust it, (7) launch Copilot from the target project directory, default automated handoff sessions to the current terminal with non-blocking approvals (`--allow-all`) to reduce trust/tool approval stalls, preserve an explicit opt-out for users who want interactive approval prompts, and provide an explicit detached-window option for users who want a separate shell, (8) instruct Squad to run the full Spec Kit lifecycle in order — `specify`, then an explicit clarify gate, then `plan`, `tasks`, `implement` — instead of relying on the human developer to manually invoke each phase, (9) default that clarify gate to `speckit.clarify` for `new-feature` and `brownfield-new` runs unless the generated spec is already materially complete for planning and Squad records a concrete skip rationale in the run log or decisions ledger, (10) tell Squad to answer clarification prompts autonomously when enough information already exists in repo context or current artifacts, (11) escalate only unresolved or high-impact clarification questions to the human developer, and (12) continue automatically through implementation once the spec/design is sufficiently clear. If the local environment cannot auto-launch Copilot/Squad, `specrew start` MUST produce an exact handoff prompt and resume-safe context, including the intended approval mode, so the developer can enter Squad with minimal manual setup while preserving the same lifecycle contract.
- **FR-025**: When `specrew start` runs in a repository that already contains a Specrew-managed Squad roster, the system MUST detect that roster as operational state rather than generic Squad bootstrap state. It MUST preserve the existing baseline roles and supplemental members, use them for lifecycle routing, and MUST NOT silently recast or replace them with a fresh generic team proposal.
- **FR-026**: When multiple delegated agents such as Claude and Codex are enabled and role policy marks lifecycle work as eligible for delegated execution, the system MUST route that work to the configured delegated agents instead of silently keeping all work on the default fast model path. Review-heavy work SHOULD prefer Reviewer/Spec Steward delegated routing, and planning/problem-solving-heavy work SHOULD prefer Planner/Spec Steward delegated routing when enabled agents make that possible. If a delegated assignment cannot be honored, the system MUST record and surface the fallback reason in the lifecycle log or decisions ledger.
- **FR-027**: The system MUST implement dynamic reasoning escalation for repeated governance-gate failures on the same artifact. It MUST track failure counts per artifact, start with an efficiency-optimized repair tier, escalate to a balanced reasoning tier after the first failed repair cycle, escalate to a deep-reasoning tier with an independent owner after the second failed repair cycle, record each escalation and owner change in the decisions ledger, and automatically de-escalate to the efficiency tier after the gate passes. When an approved Senior same-specialty pair member exists for the failing area, deep escalation SHOULD prefer that Senior pair member as the independent owner before falling back to a different independent reviewer. See `contracts/iteration-artifacts.md` for the canonical escalation state machine and required transitions.
- **FR-028**: When `specrew start` classifies a repository as `brownfield-new`, the system MUST perform a brownfield discovery pass before open-ended feature intake. That discovery MUST inspect the existing repo structure, key manifests, prominent code-language signals, Markdown/docs files, and recent git history to reconstruct a working baseline of the current product/system. The generated handoff context MUST serialize inferred technology signals, domain signals, and suggested supplemental specialists. Squad MUST use that discovery to seed or update the starting spec context, then negotiate any missing change intent through targeted interactive questions rather than treating discovery alone as sufficient scope. Brownfield starts MUST continue through the same clarify, team-shaping, team-presentation, and explicit implementation-start gates as greenfield work, and when obvious expertise gaps exist, Squad MUST propose concrete team additions aligned to the detected stack/domain instead of asking a generic open-ended specialist question.
- **FR-029**: When `specrew start` runs in `greenfield-new` mode without a supplied feature request, the system MUST actively initiate intake with an explicit interactive question about what to build and MUST remain in intake mode until it has enough grounded scope to begin `speckit.specify`. It MUST NOT merely print advisory text, make a best guess silently, or terminate as blocked after a single unanswered prompt when the user is still engaged in the session.
- **FR-030**: After `speckit.specify` and the clarify gate establish the working feature scope, the system MUST analyze the feature requirements, inferred technology choices, and current roster to determine whether additional specialist team members would materially improve delivery quality. It MUST preserve all Specrew baseline roles and user-added supplemental members, propose any missing best-fit specialists tied to the feature/technology context, and present the resulting team composition clearly to the human developer before implementation starts.
- **FR-031**: Before invoking `speckit.implement` for a new feature, the system MUST present a pre-implementation readiness handoff summarizing the ready spec/design state and the final working team, then ask the human developer for explicit approval to start implementation. Resumed in-progress features may skip this confirmation only when implementation was already previously authorized.
- **FR-032**: The system MUST encode project quality expectations as requirement-driven governance rather than a fixed universal checklist. During planning and review, it MUST determine which quality attributes are materially required for the current project/feature — for example clean code, SOLID-friendly separation of concerns, semantic correctness, concurrency safety, retries, idempotency, robustness, explicit error handling, logging, telemetry, security, and operational clarity — and MUST require plans, tasks, and review artifacts to address and validate the applicable attributes.
- **FR-033**: The system MUST discourage ceremonial sophistication in downstream implementations. If a protocol field, resiliency mechanism, architectural abstraction, or quality claim is present in the generated design or code, review/governance MUST require that it have enforced runtime semantics, validation coverage, or an explicit documented deferral. The system MUST flag unused conflict fields, fake idempotency markers, misleading recovery paths, or other sophistication-without-semantics patterns as quality defects. This rule also applies to reviewer artifacts: explicit `none`, `unknown`, `unscanned`, `not_executed`, or similar markers are acceptable when evidence is absent, but fabricated measurements, fabricated zero-findings claims, or invented structural content are quality defects.
- **FR-034**: When a feature run reaches completion, the system MUST produce a developer-facing implementation briefing before final handoff. That briefing MUST include: (1) a summary of what was built, (2) how the delivered behavior maps back to the active requirements, (3) the primary happy path and any important alternative flows, (4) the dependencies/packages the implementation uses with an explicit note on which ones are newly introduced to the project, (5) the testing strategy that was used, and (6) an explicitly labeled estimate of test coverage or confidence when exact measured coverage is unavailable. The briefing SHOULD be concise enough for a human developer preparing for code review.
- **FR-035**: System MUST provide a standalone `specrew update` CLI/script that upgrades Specrew-managed runtime surfaces in an already bootstrapped project without requiring users to infer that re-running `specrew init` is the maintenance path. Bare `specrew update` MUST update Specrew only and MUST also notify the user when newer Squad and/or Spec Kit versions are available. Latest-version discovery MUST use a canonical source per platform so downstream implementations remain consistent.
- **FR-036**: `specrew update` MUST support a non-mutating `--info` mode that reports the current installed version and the latest known available version for Specrew, Squad, and Spec Kit. When a latest-version check cannot be completed, the command MUST say so explicitly rather than pretending the platform is current. In v1, the canonical latest-version sources are: Specrew releases/tags from the Specrew upstream GitHub repository, Spec Kit release tags from the official Spec Kit upstream GitHub repository, and the npm registry for Squad.
- **FR-037**: `specrew update` MUST support explicit scope control for dependency upgrades. It MUST allow users to request Specrew-only updates (default), all-platform updates (`--all`), and platform-specific updates for Specrew, Squad, and Spec Kit. When updating Squad and/or Spec Kit, the command MUST validate compatibility, preserve user-owned downstream artifacts, and report the result independently for each requested platform.
- **FR-038**: After `speckit.specify` and the clarify gate establish the working feature scope, the system MUST perform a concurrency-aware team-sizing analysis in addition to the existing missing-specialist analysis. That analysis MUST consider the current roster, inferred technology choices, the current plan/task dependency graph, likely workstream separability, dependency constraints, shared-surface conflict risk, implementation risk, and when prior reviewer artifacts exist, file-ownership and hotspot signals from the most recent code map. The system MUST record an auditable concurrency rationale in the active `plan.md` before proposing same-specialty capacity expansion.
- **FR-039**: When the concurrency analysis indicates that same-specialty capacity expansion is warranted, the system MUST be able to propose paired same-specialty team members using the default terminology **Junior** and **Senior** for that specialty (for example Junior Frontend Developer + Senior Frontend Developer). Specrew MAY allow downstream projects to override those default labels via `role-assignments.yml`, provided the resulting roles preserve the same routing semantics, distinct identity, and approval requirements. These paired roles MUST be presented as distinct named team members with separate charters, routing intent, and model/capability profiles rather than anonymous duplicate copies of a single role identity. The proposal MUST explain why the pair is needed and MUST require explicit human approval before implementation begins.
- **FR-040**: When an approved Junior/Senior same-specialty pair exists, the runtime routing policy MUST distinguish their task classes. Junior roles SHOULD receive bounded, lower-risk, well-scoped implementation slices optimized for faster execution. Senior roles SHOULD receive ambiguous, cross-cutting, integration-heavy, security-sensitive, concurrency-sensitive, or reviewer-gated work. The coordinator MAY run Junior and Senior same-specialty work in parallel only when ownership boundaries and task dependencies are explicit enough to avoid redundant or conflicting execution.
- **FR-041**: Governance for same-specialty parallel execution MUST protect against unsafe parallelism. Before launching concurrent Junior/Senior execution within the same specialty, the system MUST define explicit task ownership boundaries or a coordination plan, MUST avoid assigning overlapping logical slices without that plan, and MUST support escalation from Junior-owned work to Senior-owned work when repeated governance failures, integration risk, or review policy indicate that deeper oversight is required. In v1, explicit boundaries MUST be recorded in `tasks.md` as `owner_file_globs` (or an equivalent serialized ownership-boundary field) for any tasks allowed to run in parallel; if those boundaries are missing, runtime policy MUST keep the work serial.
- **FR-042**: In a later iteration, the system MUST provide a staged multi-lane validation strategy for Specrew itself. The primary PR gate MUST remain deterministic and artifact-based, covering governance validation and integration flows without requiring live agent execution. A second contract lane MUST validate generated prompts, handoff context, routing decisions, and structured lifecycle traces against policy expectations. A third confidence lane MUST support a small set of live Copilot/Squad smoke scenarios on a scheduled or pre-release basis rather than as mandatory per-PR checks, and those live runs MUST persist structured traces so important failures can be replayed later as deterministic regression fixtures.
- **FR-043**: When delegated routing is active for any lifecycle, review, or repair work item, the system MUST record visible runtime-routing evidence. That evidence MUST include the role or work item, requested agent family, effective agent family, concrete model ID, and whether the delegated assignment was honored or fell back. When a delegated assignment is not honored, the fallback reason MUST be recorded in the same evidence trail. Runtime-routing evidence writes MUST be performed by Specrew-owned scripts at the time routing/escalation decisions are applied; delegated coordinator instructions alone are not sufficient evidence.
- **FR-044**: The system MUST enforce a no-gap iteration exit policy for lifecycle-governed work. If review, governance validation, or runtime evidence reveals a known alignment gap across spec, implementation, tests, docs, or observability, the run MUST NOT close as complete unless the gap is either repaired in the current iteration or explicitly deferred with human approval. Any approved defer MUST be recorded canonically in `.squad\decisions.md` as a defer entry, including the affected requirement or artifact, rationale, approving human, and next action, and linked from any affected review or planning artifact as needed so the gap does not roll silently into the next iteration.
- **FR-045**: The system MUST provide a critical, evidence-driven review mode for governance-heavy work. During review and final readiness checks, the reviewer SHOULD classify each hardened lifecycle/governance requirement against four dimensions — implemented, enforced, observable, and documented — and MUST emit a gap ledger whenever any dimension is missing. In v1, the canonical gap-ledger surface is a `## Gap Ledger` section inside `review.md`, and the reviewer index MUST mirror its active concerns into triage hints. If that review finds an ambiguity, contradiction, or missing decision in the governing spec, the system MUST stop closure, ask the human developer targeted clarification questions, update the spec with the answer, and reconcile any affected plan/task/governance artifacts before continuing.
- **FR-046**: The system MUST generate `code-map.md` at iteration close for every iteration that touches code. The artifact MUST include a Files Touched table (path, lines added, lines removed, owning task ID(s), owning role), a Public-API Delta section listing best-effort added/removed top-level exports or equivalent language constructs, a Module Hotspots section using thresholds from `iteration-config.yml`, and a Test-to-Code Ratio line computed from reviewer test-path heuristics in `iteration-config.yml`. Specrew MUST record the iteration baseline ref at iteration start in `state.md`, and reviewer artifacts in FR-046 through FR-049 MUST compute their diffs against that baseline ref. It MUST behave as an index and MUST NOT copy source excerpts into the artifact.
- **FR-047**: The system MUST generate `dependency-report.md` at iteration close for every iteration that modifies a manifest file. The artifact MUST include per-ecosystem dependency deltas (package, prior version, new version, change type, declared license if available, owning task), a New-to-Project subsection for first-introduced packages, a Vulnerability Scan subsection that either embeds recognized external scanner output with tool name/version/exit code or explicitly records `unscanned` with the reason, and a Transitive Surface note describing whether transitive dependencies were resolved and by which command. License fields MUST come from manifest or lockfile metadata when available, and MUST be marked unknown when unavailable. In v1, auto-detected vulnerability scanning MUST be limited to commands listed in `reviewer.vulnerability_scanner.candidates` in `iteration-config.yml`.
- **FR-048**: When the plan declares security as a materially required quality attribute under FR-032, or when the team includes a security-focused role, the system MUST generate `security-surface.md` at iteration close. The artifact MUST enumerate trust boundaries touched, sensitive-data touchpoints matched from `iteration-config.yml` pattern sets, any security-specialist findings when such a specialist is on the team, and any HIGH/CRITICAL dependency findings reproduced from the dependency report. When the trigger conditions are not met, the artifact MUST be omitted and the reviewer index MUST explicitly state why it was not generated.
- **FR-049**: The system MUST generate `coverage-evidence.md` at iteration close for every iteration that touches code. The artifact MUST include a referenced test-strategy section tied back to the implementation briefing, a Tests Run table with actual command, pass/fail counts, duration, and exit code from review-time execution of test commands declared in `iteration-config.yml`, a Coverage Estimate section that records either measured coverage (with tool, coverage kind, and percentage) or an explicitly labeled qualitative estimate, and a Coverage-to-Requirements mapping derived from task-to-requirement and task-to-test traceability. When `test_commands` is empty, `reviewer.skip_test_execution_at_close` is enabled, or review-time execution is unsafe/unavailable, Specrew MAY skip execution, but it MUST record `not_executed` with a reason and the reviewer summary/digest MUST surface that outcome explicitly rather than implying a passing test run or measured coverage.
- **FR-050**: When an iteration closes in an interactive terminal, the system MUST automatically emit a visually delimited Reviewer Summary block to stdout as the primary reviewer-entry surface. The block MUST cover, in stable order: header (iteration, feature, branch, commit range), verdict, requirements covered/not covered, code surface, dependencies, coverage, operational signals (escalations and routing fallbacks), drift, reviewer-index pointer, implementation-briefing pointer, and local-open hints for the primary reviewer artifacts when available. The summary MUST remain readable in plain terminals and support an ASCII-safe fallback when `SPECREW_PLAIN=1` is set. The implementation SHOULD keep the summary within roughly 40 lines on a conventional 80x24 terminal so the closeout stays scannable.
- **FR-051**: When iteration close runs in non-interactive mode (`--quiet`, redirected output, or `$env:CI`), the system MUST emit exactly one stable machine-parseable reviewer digest line to stdout and MUST suppress the visual Reviewer Summary. The digest MUST begin with `SPECREW_REVIEW schema=v1`, and the remaining fields, order, and tokens MUST remain stable so CI consumers can parse them without ambiguity.
- **FR-052**: The system MUST generate `reviewer-index.md` at iteration close as the single reviewer entrypoint, containing the same headline summary as the interactive Reviewer Summary, a recommended read order, links to related iteration artifacts and decisions evidence, and triage hints for hotspots, vulnerability findings, unknown licenses, escalation events, unresolved drift, and gap-ledger carry-forwards. The system MUST also provide a standalone `specrew review [<iteration>]` command as a replay/reopen surface for the requested or most recently closed iteration, re-emitting the persisted reviewer summary without mutating iteration state, with support for `--quiet`, `--json`, `--feature <id>`, and `--open`. In v1, `--open` MUST attempt to open `reviewer-index.md` in the operating system's default local handler and also open `review-diagrams.md` when present; if local opening is unavailable, Specrew MUST print the relevant paths explicitly instead of failing silently.
- **FR-053**: The system MUST support diagram-first reviewer navigation using Mermaid as the default reviewer diagram format in v1. For code-touching iterations, it MUST generate `review-diagrams.md` when deterministic structure or flow thresholds are met, using Mermaid blocks to summarize module relationships and, when materially changed, important runtime or request flows. In v1, the thresholds MUST come from `iteration-config.yml` (for example structure modules touched and inter-module edges, plus flow entrypoint and module-count thresholds). The reviewer summary and reviewer index MUST link to this artifact and MUST support local-open viewing surfaces (such as relative path, `file:///` path, or equivalent local viewer invocation) without requiring any third-party hosted rendering service. If Specrew lacks enough grounded evidence to generate a meaningful diagram, it MUST explicitly omit the diagram or record why the diagram is unavailable rather than inventing one.
- **FR-054**: Reviewer artifacts under `specs/<feature>/iterations/<NNN>/` MUST be immutable iteration snapshots once that iteration closes. Later iterations MUST NOT rewrite prior iteration reviewer artifacts to reflect newer code state. If Specrew provides a current-view surface for reviewer orientation, it MUST keep that as a separate mutable artifact outside the closed iteration directory (for example `specs/<feature>/current-architecture.md`) and the reviewer index MUST clearly distinguish snapshot artifacts from current-view artifacts.
- **FR-055**: `specrew init` MUST scaffold a downstream repository hygiene contract. That contract MUST classify generated files into at least shared tracked governance state versus per-machine transient runtime state, MUST install or update a downstream `.gitignore` so transient Specrew runtime artifacts are ignored by default, and MUST generate a `docs/repo-hygiene.md` reference that explains the supported categories. In that contract, `.squad/agents/<role>/history.md` MUST be treated as per-machine local notebook state rather than shared project truth, and the documentation MUST explicitly state that team-binding facts belong in `.squad\decisions.md` or tracked iteration artifacts instead. The transient-runtime list MUST include, at minimum: `.specrew\last-start-prompt.md`, `.specrew\start-context.json`, `.specrew\start-summary.md`, `.specrew\bootstrap-dry-run-*.md`, `.scratch\`, `.squad\.scratch\`, `.squad\orchestration-log\`, `.squad\log\`, `.squad\sessions\`, `.squad\decisions\inbox\`, `.squad\agents\<role>\history.md`, `.worf-review\`, `.squad-workstream`, and `*.backup`. The contract MUST preserve cross-machine continuation by keeping resume-critical shared artifacts — including `specs/<feature>/iterations/<NNN>/state.md`, tracked governance/config files, and shared decisions evidence — under version control so another machine can continue the same iteration from the last committed checkpoint.
- **FR-056**: At governance boundaries, Specrew MUST support an explicit commit-offer workflow without silently auto-committing. After presenting the closeout review for an iteration, the system MUST be able to offer the user a commit covering the completed iteration before the run is fully handed off. Before the first file change of a subsequent iteration, if tracked changes from the previous iteration are still uncommitted, the system MUST be able to offer a commit for those pending changes before proceeding. Any such commit MUST require explicit user approval; silent auto-commit is not allowed. If the user declines, Specrew MUST continue safely while warning that cross-machine continuation and destructive git recovery only preserve the last committed checkpoint.

### Non-Goals

- **NG-001**: Specrew does NOT build a new coding-agent runtime. It relies on Squad.
- **NG-002**: Specrew does NOT replace Spec Kit or Squad. It extends both.
- **NG-003**: Specrew does NOT provide a graphical dashboard or UI in the initial version. Governance visibility is through spec artifacts and iteration reports.
- **NG-004**: Specrew does NOT require non-Copilot execution paths in v1. Specrew does NOT track or display billing/cost context for delegated agents; that surface is GitHub's. Copilot remains the default workhorse outside delegated routing and escalation policy (FR-021, FR-022, FR-026).
- **NG-005**: Specrew does NOT enforce a specific programming language, framework, or project type for downstream projects.
- **NG-006**: Specrew does NOT manage CI/CD pipelines. It may produce artifacts that CI/CD consumes, but pipeline configuration is out of scope.
- **NG-007**: Specrew does NOT attempt to resolve all drift automatically. It surfaces drift and proposes options; the human decides.

- **NG-010**: Specrew does NOT report token, dollar, or billing cost in v1. Reasoning-tier escalation history is the only cost-adjacent operational signal exposed, and it MUST NOT be presented as a numeric cost claim.
- **NG-011**: Specrew does NOT require a third-party hosted diagram-rendering service in v1. Reviewer diagrams MUST remain usable from local Markdown, local browser viewing, or editor-native preview surfaces.

### Downstream User Flow

After `specrew init`, the canonical downstream entrypoint is `specrew start`. Specrew still relies on Spec Kit and Squad as the underlying surfaces, but the human developer no longer has to manually drive each Spec Kit phase command in sequence.

| Step | Surface | Action | Specrew Governance |
| ---- | ------- | ------ | ------------------ |
| 1 | CLI | `specrew init` | Bootstrap: scaffold config, install extensions, assign roles |
| 2 | CLI | `specrew start` | Canonical entrypoint. Launch or hand off to Squad with lifecycle context, resume active work, or gather the next feature/fix interactively |
| 3 | Squad + Spec Kit | intake / `specify` | Squad negotiates greenfield scope as needed, then creates or updates the feature spec |
| 4 | Squad + Spec Kit | `clarify` | Squad answers its own questions when possible; unresolved questions go to the human developer |
| 5 | Squad | team shaping | After spec/clarify, Squad proposes missing specialists, analyzes safe delivery concurrency, may propose Junior/Senior same-specialty pairs when warranted, preserves user-added members, and presents the final team |
| 6 | Squad + Spec Kit | `plan` | Create implementation plan. `before_plan` hook validates spec has requirements |
| 7 | Squad + Spec Kit | `tasks` | Generate tasks. `after_tasks` hook verifies traceability and quality coverage |
| 8 | Squad | implementation handoff | Squad asks the human developer to start implementation once spec and team are ready |
| 9 | Squad + Spec Kit | `implement` | Execute implementation after explicit go-ahead |
| 10 | Squad | Review/Demo ceremony | Per-task verdicts. Drift-check skill runs as batch fallback if not per-task |
| 11 | Squad | gap-ledger review | Reviewer classifies hardened requirements as implemented / enforced / observable / documented, records gaps, and routes spec ambiguities into clarification before closure |
| 12 | Squad | implementation briefing | Squad presents a code-review-ready summary of behavior, requirements, flows, dependencies, and testing confidence |
| 13 | Specrew | reviewer artifact generation + console summary | Specrew generates reviewer artifacts (`code-map.md`, `dependency-report.md`, `coverage-evidence.md`, conditional `security-surface.md`, `reviewer-index.md`, `review-diagrams.md` when grounded), automatically emits the reviewer summary/digest at close, and includes local-open hints for the primary review surfaces |
| 14 | Squad | Retrospective ceremony | Retro Facilitator generates `retro.md` artifact |
| 15 | CLI | `specrew start` | Repeat the lifecycle for the next feature or fix, with Squad gathering intake when needed |

#### Specrew v1 Design Decisions

### Key Entities

All iteration artifacts are stored as Markdown files within the spec feature directory (e.g., `specs/001-feature/iterations/001/`), following Spec Kit conventions for human readability and git-diffability.

**Delegated Agent**: A Copilot-accessible agent option available to Specrew (Copilot default, Claude, Codex). Each option has an access path and an availability state (detected/enabled/disabled). Squad still runs on Copilot; Specrew configures which agent Copilot should use for selected roles. Billing/cost context is out of scope for Specrew — any cost implications are managed by the user through GitHub.

- **Capacity-Tier Pair**: An optional same-specialty team expansion approved after spec/clarify when Specrew detects safe, meaningful parallel work inside one specialty. In this feature, the pair is expressed as **Junior** and **Senior** variants of the same specialty. Junior is optimized for bounded, lower-risk execution slices; Senior is optimized for ambiguous, cross-cutting, or higher-risk slices and may also serve as an escalation path for Junior-owned work.

- **Specrew Configuration**: The bootstrap-generated state that ties the Spec Kit extension and Squad extension together for a given project. Includes iteration settings, role assignments, and governance artifact locations.
- **Downstream Constitution**: A governance template generated for the user's project, distinct from Specrew's own project constitution. Customizable by the user.
- **Iteration**: A time-boxed or scope-boxed delivery cycle with four phases (planning, execution, review/demo, retrospective). Scoped to a single spec.
- **Iteration Plan**: A set of tasks for one iteration, each mapped to a spec requirement, with effort estimates, assigned owners, and traceability links.
- **Spec Steward**: A dedicated role (assigned to an AI agent, a human, or both) responsible for monitoring alignment between spec, plan, tasks, decisions, and implementation.
- **Drift Event**: A recorded instance where implementation or agent behavior diverges from the spec. Includes the specific requirement, the deviation, and the resolution path chosen.
- **Evaluation Report**: The output of the evaluation harness. Contains separate assessments for process quality and outcome quality across one or more iterations.
- **Collision Record**: A detected conflict between Specrew and another extension, including the conflicting resource, the involved extensions, and the resolution options presented to the user.
- **Crew Composition**: The set of agent roles active in a project. The baseline consists of five protected roles (Spec Steward, Planner, Implementer, Reviewer, Retro Facilitator) deployed deterministically during `specrew init` using Squad's non-interactive mode — bypassing Squad's team-member casting interview to ensure bootstrap completes without blocking. The Retro Facilitator provides Specrew-specific governance prompts (estimation accuracy, drift summary, process adherence, improvement actions) and owns `retro.md` generation — Squad's built-in Retrospective ceremony provides the infrastructure. After bootstrap, users manage domain-specific members (e.g., Security Analyst, UX Designer, DBA) through Specrew's command-driven team interface (`specrew team add/update/remove`), which ensures all required Squad artifacts are created, updated, or removed atomically without manual multi-file editing. Baseline roles are protected (cannot be removed), only supplemented.
- **Materially Complete (for planning)**: A spec is materially complete for planning when it already contains enough accepted scope, constraints, and success conditions for the Planner to generate a non-speculative task plan without needing additional product-defining clarification.
- **Grounded Scope / Grounded Evidence**: Scope or evidence is grounded when it is supported by explicit user answers, tracked artifacts, or repository evidence rather than agent guesswork. A grounded gate means Specrew has enough cited input to proceed without inventing missing product intent.
- **Review-Heavy Work**: Lifecycle work dominated by review/demo, governance validation, drift assessment, security review, or other independent-verification responsibilities rather than primary implementation.
- **Problem-Solving-Heavy Work**: Lifecycle work dominated by planning, architecture tradeoffs, ambiguity reduction, debugging, root-cause analysis, or other high-judgment reasoning tasks rather than straightforward execution slices.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A new project can be bootstrapped with Specrew (both extensions installed, governance artifacts scaffolded, Squad team configured) in under 10 minutes of user effort.
- **SC-002**: An AI crew can run at least 3 consecutive iterations against a spec, producing working deliverables traced to requirements in each iteration.
- **SC-003**: Spec drift is detected and surfaced to the user within the same iteration it occurs — not discovered only during final review.
- **SC-004**: The evaluation harness can run against a reference spec, complete at least 2 iterations, and produce a structured pass/fail report in a single automated session.

- **SC-006**: Iteration capacity overcommitment is flagged before execution begins in at least 90% of cases.

- **SC-008**: Specrew coexists with at least one other Spec Kit extension and one other Squad plugin in the same project without interference.
- **SC-009**: Downstream governance artifacts are clearly distinguishable from Specrew's own governance in 100% of bootstrapped projects.

## Assumptions

- Specrew owns the dependency lifecycle via `specrew init`. Users do not need to pre-install Spec Kit or Squad — Specrew installs them at pinned compatible versions if not already present.
- The execution runtime for v1 is GitHub Copilot via Squad. Additional agents may be used only as Copilot Agent HQ delegated options; Specrew does not invoke standalone Claude/Codex runtimes in v1.
- The Spec Steward role can be assigned to an AI agent, a human, or a combination — the assignment is made by the user during bootstrap and can be changed at any time.
- Iteration length is configurable. The default is scope-boxed (a set of tasks) rather than time-boxed, because AI crews do not have fixed working hours.
- The evaluation harness will use a controlled reference spec with predetermined requirements. It does not need to evaluate arbitrary user specs.

## Platform Facts vs. Specrew v1 Design Decisions

The following are **documented platform capabilities** confirmed in public Spec Kit and Squad sources:

- Spec Kit provides a structured flow: install → init → constitution → specify → clarify → plan → tasks → implement.
- Spec Kit’s extension model supports namespaced commands, lifecycle hooks, config layering, and template presets.
- Squad currently supports GitHub Copilot as its coding-agent runtime.
- Squad’s extension model packages reusable behavior as skills, ceremonies, and directives.
- Squad ships with built-in **Design Review** and **Retrospective** ceremonies.

The following are **Specrew v1 design decisions** that build on those platforms but are not externally documented platform facts:

- Pinned minimum versions: Spec Kit >= 0.8.4, Squad >= 0.9.1. *(Chosen based on versions current at Specrew design time. Subject to revision as both platforms evolve.)*
- Five protected baseline roles (Spec Steward, Planner, Implementer, Reviewer, Retro Facilitator) deployed deterministically via `squad init --non-interactive` during bootstrap, bypassing Squad's team-member casting interview to ensure bootstrap completes without user interaction or blocking. Post-bootstrap team management is command-driven via `specrew team add/update/remove` commands that atomically create, update, or delete all required Squad artifacts (`.squad/team.md`, `.squad/agents/*/charter.md`, `.squad/agents/*/history.md`) without manual multi-file editing. *(A Specrew product choice, not a Squad-mandated default.)*
- Story points as the default effort unit, with configurable alternatives. *(A Specrew default, not a platform-level concept.)*
- Active post-task drift detection via a Squad post-task hook. *(Specrew’s intended use of Squad’s hook pipeline. The exact hook shape must be validated against Squad’s current SDK surface during implementation.)*
- Planning and Review/Demo as Specrew-defined ceremonies, distinct from Squad’s built-in Design Review and Retrospective. *(Specrew adds these; it does not restate existing Squad behavior.)*
- The Specrew project itself MUST be managed via a private GitHub repository and GitHub Projects V2. *(This is a normative requirement for Specrew's own development. Downstream projects MAY choose whether to use a GitHub Project board; they are not required and may use Jira, Azure DevOps, GitHub Projects, or local tasks.md.)*
- Specrew uses the official Spec Kit extension starter template for its Spec Kit extension and Squad's documented extension structure (skills/, ceremonies/, directives/, README.md) for its Squad runtime-surface deployment. *(No third-party scaffolding tools for runtime architecture.)*
- Copilot remains the only Squad runtime in v1; Claude/Codex, when available, are optional delegated agents selected through Copilot Agent HQ for review roles. *(This preserves Squad's execution model while enabling independent review perspective.)*
- Squad is used to develop Specrew itself from Iteration 0 — dogfooding from day one. *(Ensures the product is validated against its own development process.)*
- Local task artifacts (plan.md, task lists, iteration state) are the authoritative source of truth for Specrew planning. GitHub Issues and Project board items are created by Squad automation for visibility and operational tracking. *(For downstream projects, the choice of which system is authoritative is made by the project.)*
- Human review of Specrew is handled through standard GitHub PR review. No formal "Human Reviewer" crew role is assigned. *(Keeps the crew model agent-focused; human oversight occurs through normal PR and iteration approval gates.)*
- Squad is responsible for Specrew self-development board sync and maintenance: creating GitHub Issues from authoritative plan/task artifacts, adding issues to the board, maintaining status through execution/review/retro phases, and reflecting closure on the board. *(Manual board management is fallback-only if automation fails; it is NOT the normal operating rule. If Squad cannot populate/manage the board, this MUST be recorded as a capability gap/blocker rather than silently degrading to manual management. No Spec Kit-side project-management extension is needed.)*
- `specrew init` is a standalone CLI/script at the monorepo root — not a Spec Kit extension command. It must work before `.specify/` or `.squad/` exist. It calls `specify init` and `squad init` as sub-steps, then installs the Specrew Spec Kit extension and deploys Squad skills/ceremonies/directives into Squad's native runtime locations. *(The Spec Kit extension activates only after `.specify/` exists; Squad skills/ceremonies activate only after `.squad/` exists. `specrew init` is the orchestration layer above both.)*
- Specrew's GitHub Projects V2 board uses Squad's documented default board layout without custom columns. The default layout MUST support mapping iteration phases (planning → executing → reviewing → retro → complete) to board state transitions driven by automation. *(No custom "Review" column or other board customizations are required.)*

## Governance Alignment *(mandatory)*

- **Spec Steward**: Assigned by the user during bootstrap. Can be an AI agent role, a human, or both. Responsible for spec integrity across all phases.
- **Iteration Facilitator**: The user or an assigned human who starts iterations, approves plans, and participates in review/demo/retro. May be the same person as the Spec Steward for solo users.
- **Capacity Model**: Effort units per iteration (Specrew v1 default: story points, configurable). Default iteration is scope-boxed. Capacity is set by the user and calibrated by retro data over time.
- **Drift Signals**: The Spec Steward monitors task outputs against source requirements during execution. The review/demo gate performs a formal alignment check. Drift events are recorded with citations.
- **Human Oversight Points**: (1) Iteration plan approval before execution starts, (2) review/demo verdicts before an iteration can enter retrospective closure, (3) Alon sign-off after `retro.md` exists and before the iteration is marked `complete` or the next iteration begins, (4) any drift resolution that involves changing the spec.
- **Constitution Stability**: The project constitution is intended to remain stable and principle-oriented. Product behavior, defaults, and operational detail belong in the spec and plan — not the constitution. Future detail growth should be directed to spec/plan artifacts rather than amending the constitution.

## Iteration Lifecycle Contract *(normative)*

Every iteration MUST follow a strict four-phase state machine. This is not optional governance — it is a binding operational rule that Specrew itself enforces and follows.

### Phase State Machine

```text
┌─────────┐       ┌───────────┐       ┌────────────┐       ┌──────────┐       ┌──────────┐
│Planning │──────>│ Executing │──────>│ Reviewing  │──────>│  Retro   │──────>│ Complete │
└─────────┘       └───────────┘       └────────────┘       └──────────┘       └──────────┘
      │                  │                    │                  │
      └──────────────────┴────────────────────┴──────────────────┘
                         (Abandoned)
```

**Mandatory Transitions**:

- **planning → executing**: Iteration plan MUST be approved by user/Spec Steward before any task execution begins.
- **executing → reviewing**: All planned tasks MUST be completed, abandoned, or deferred before review gate opens.
- **reviewing → retro**: Review verdicts MUST be recorded per task and iteration-level. If verdict is "needs-rework", affected tasks re-enter executing phase.
- **retro → complete**: Retrospective MUST produce `retro.md` with estimation accuracy, drift summary, and improvement actions. The iteration remains in `retro` until Alon records final sign-off; only then may it transition to `complete`.
- **any phase → abandoned**: Explicit reason MUST be recorded. Incomplete tasks become available for next iteration.

**Artifact Production per Phase**:

- **planning**: `plan.md` (required) — tasks, effort estimates, owners, traceability to requirements
- **executing**: `state.md` (required) — task completion state, resume tokens; `drift-log.md` (required) — all drift events detected
- **reviewing**: `review.md` (required) — per-task verdicts, overall iteration verdict
- **retro**: `retro.md` (required) — estimation accuracy, drift summary, process notes, improvement actions

**Violation Triggers**:

- Skipping a phase (e.g., executing → complete without review) is a contract violation. Next iteration cannot begin.
- Missing required artifacts blocks phase transition. Governance validator (FR-012) enforces this at runtime.
- Attempting to approve a plan without spec requirements or attempting to execute without an approved plan is an error.

## Dogfooding Obligation *(mandatory)*

Specrew MUST follow its own governance model for its own development. This is not a recommendation — it is a binding obligation that ensures:

1. **Specrew uses Specrew**: The Specrew project itself is developed using the four-phase iteration lifecycle. Every iteration (including development of this product) goes through planning → execution → review/demo → retrospective.

2. **Specs are authoritative**: Specrew's own feature specs (`specs/001-specrew-product/spec.md`, etc.) are the source of truth for Specrew development. No implementation diverges without a tracked change to the spec.

3. **Drift detection applies to Specrew**: Drift detection (FR-008) MUST be active during Specrew's own development. After each task in a Specrew iteration, the Spec Steward checks alignment to the source requirement. Drift is surfaced and resolved the same way as downstream projects would handle it.

4. **Full artifact lifecycle**: Every Specrew iteration MUST produce all four-phase artifacts (`plan.md`, `state.md`, `drift-log.md`, `review.md`, `retro.md`). These artifacts are version-controlled in the Specrew repository and available for audit.

5. **Implications**:
   - Specrew development iterations may NOT skip the retrospective phase.
   - Iteration plans for Specrew development MUST map every task to a feature requirement (FR or TG).
   - Review verdicts MUST be recorded per task against the originating requirement.
   - No feature or iteration is considered `complete` for Specrew until the retrospective artifact exists and Alon sign-off is recorded.

**Exceptions**: Architecture spikes or infrastructure work that is not mapped to a user-visible feature may be grouped under a support/infrastructure FR for planning purposes, but MUST still be traceable and MUST still undergo review/retro.
