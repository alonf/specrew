# Contract: Architecture Intent Brief Schema

**Contract Version**: 1.0.0  
**Format**: JSON Schema (for Phase 2 machine consumption); Markdown template (Phase 1 human presentation)  
**Purpose**: Define the structure and validation rules for architecture intent briefs

## Overview

This contract specifies two representations of the architecture intent brief:

1. **Markdown Template** (Phase 1): Human-readable format used in chat/CLI presentation
2. **JSON Schema** (Phase 2): Machine-readable format for automated validation and enforcement

## Part 1: Markdown Template (Phase 1)

### Structure

```markdown
# Architecture Intent Brief: [FEATURE_ID]

**Feature**: [FEATURE_ID]  
**Generated By**: Squad Copilot  
**Generated At**: [ISO 8601 DATETIME]  
**Status**: [pending_review | approved | rejected]

## Design Approach

[1–2 paragraph summary of the intended architecture or design approach. Clear, non-technical language where possible. Examples of design approach:
- "Extend the existing API middleware to support the new authentication method"
- "Create a new microservice for real-time notifications; integrate via event bus"
- "Use embedded database (SQLite) for the new offline-sync feature; sync to backend on reconnect"]

## Affected Surfaces

- **[Surface 1]** (e.g., `src/api/auth/`) — Justification: [Why this surface will change and how materially it will be affected]
- **[Surface 2]** (e.g., `database/schema/`) — Justification: [Why this surface will change and how materially it will be affected]
- **[Surface N]** — Justification: [...]

*Guideline: List at least 1 surface; be specific (avoid vague surfaces like "codebase"). Include brief justification for each.*

## Expected Changes

### Data Model

[Changes to entities, schemas, migrations, or persistence model. Format as before/after or impact statement.]

Example:
```
BEFORE: User has single email address; no phone field
AFTER: User gains phone field (optional string); email becomes multi-valued
IMPACT: Existing code checking user.email must handle multi-value; migration script needed
```

### API or Public Contracts

[Changes to REST endpoints, function signatures, CLI arguments, event schemas, or other public surfaces.]

Example:
```
NEW ENDPOINT: POST /auth/verify-otp
  - Input: { user_id, otp_code }
  - Output: { token, expires_at }
  - Error: 401 Unauthorized if OTP is invalid

CHANGED ENDPOINT: POST /auth/login
  - Old: { email, password } → { token }
  - New: Supports optional { email_or_phone, password } → { requires_otp: boolean }
```

### Workflow or User-Facing Changes

[Changes to user experience, business logic, or operational workflows.]

Example:
```
BEFORE: User logs in with email/password, receives session token
AFTER: User logs in with email/password, receives OTP via SMS/email, confirms OTP, receives session token
IMPACT: Two additional round-trips for first-time login
```

### Storage or Persistence Changes

[New databases, new tables, schema migrations, or changes to data retention.]

Example:
```
NEW TABLE: otp_sessions
  - otp_code (hashed string)
  - user_id (foreign key)
  - created_at (timestamp)
  - expires_at (timestamp)
  - verified_at (null until confirmed)
  - attempt_count (integer, max 3)
```

---

## Dependencies and Tooling

[Important new frameworks, libraries, versions, or tools.]

| Dependency | Version | Why Chosen | Alternatives |
| --- | --- | --- | --- |
| [e.g., `twilio`] | [e.g., `^2.0.0`] | [e.g., industry standard for SMS/OTP delivery] | [e.g., AWS SNS, Vonage] |
| [e.g., `redis` for OTP cache] | [e.g., `^6.2.0`] | [fast in-memory store for short-lived OTP sessions] | [memcached, local cache] |

*Guideline: Only list new dependencies or significant version upgrades. Omit if no new dependencies.*

## Implications

### Security

[Security implications: new trust boundaries, auth/authz changes, credential handling, encryption, etc. If no security implications, state "None identified in this iteration."]

Example: "OTP codes are stored hashed; SMS delivery is external (Twilio). Risk: Twilio account compromise could allow OTP interception. Mitigation: Twilio API credentials stored in secrets manager; OTP attempts rate-limited."

### Performance

[Performance implications: latency, throughput, resource usage, scaling considerations.]

Example: "OTP verification adds 1 network call to Twilio; cached OTP sessions in Redis reduce database hits. Expected P95 latency: +50ms per OTP delivery request."

### Reliability and Resilience

[Failure modes, retry semantics, fallback behavior, idempotency.]

Example: "If Twilio is unavailable, login fails. No fallback. Mitigation: Twilio SLA 99.9%; document manual recovery process."

### Compatibility and Migration

[Backward compatibility, database migrations, deployment sequencing, rollback plan.]

Example: "Database migration adds otp_sessions table; safe for zero-downtime deploy. Rollback: drop table (no data loss if done <72 hours after deploy, per retention policy)."

---

## Known Assumptions

- [Assumption 1: e.g., "OTP delivery via SMS is acceptable latency for users (expect 5–30 second delivery)"]
- [Assumption 2: e.g., "Twilio service is available in all regions where we operate"]
- [Assumption 3: e.g., "Users have access to email or SMS for OTP reception"]
- [Assumption N: ...]

*Guideline: Make assumptions explicit so they can be validated or challenged. At least one assumption should be present.*

---

## Alternatives Considered

*Guidelines*: 
- Include alternatives WHEN they materially differ from the primary approach in cost, risk, or reversibility.
- For routine convention-following features with no meaningful architectural choices, state "No material alternatives; follows existing conventions" and omit detailed alternative analysis.
- When alternatives are presented, include at least one meaningful alternative with explicit trade-off analysis.

### Alternative 1: [Name of Alternative]

**Approach**: [Brief description of how this alternative would work]

**Trade-Offs**:
- ✅ **Advantages**: [e.g., simpler, no new dependencies, existing patterns in codebase]
- ❌ **Disadvantages**: [e.g., slower, less secure, requires larger refactor]

**Why Not Chosen**: [Concise reason why this was rejected in favor of the primary design approach]

### Alternative 2: [Name of Alternative]

[Same structure as Alternative 1]

### Alternative N: [...]

---

## Open Questions for Human Decision

### Question 1

**Question**: [Specific decision point; e.g., "Should OTP be delivered via SMS or email, or both?"]

**Options**:
1. **SMS only**: Fast delivery; requires phone number; costs per message (Twilio ~$0.01/SMS)
2. **Email only**: No delivery cost; slower (1–5 min typical); all users have email
3. **Both**: User chooses preferred method; more complex UI; higher operational cost
4. **Email with optional SMS**: Email by default; SMS available for urgent cases

**Squad's Recommendation**: Option 2 (Email only) for MVP based on cost and simplicity; user preference for SMS can be added in a future iteration if demand exists.

**Why This Matters**: Affects dependency choices (Twilio vs. built-in email), UX complexity, operational cost, and user privacy (phone numbers).

### Question 2

[Similar structure for other open questions]

---

## Approval Status

- **Status**: `pending_review` (awaiting human decision)
- **Waiting For**: Human architect/developer to review, ask clarifying questions, and approve, reject, or constrain the direction

---

## How to Use This Brief

1. **Review**: Read the Design Approach, Affected Surfaces, and Expected Changes to understand the proposed direction
2. **Assess**: Consider the Implications and Alternatives; ask clarifying questions if needed
3. **Decide**: For each Open Question, provide your preference or ask Squad to reconsider
4. **Record**: Once you approve the direction (with or without constraints), the decision is recorded in plan.md § Architecture Intent Review

---
```

### Validation Rules (JSON Schema Enforcement)

1. **Required Fields**: All top-level `required` fields must be present
2. **design_approach**: Must be 50–2000 characters (substantive, not placeholder)
3. **affected_surfaces**: At least 1 surface; each requires justification
4. **assumptions**: At least 1 assumption; each must be specific
5. **alternatives_considered**: May be empty for routine convention-following features; when present, at least 1 alternative with trade-off analysis required
6. **open_questions**: May be empty only if all choices are clear; each question must have at least 2 options
7. **approval_status**: Must be one of the controlled values

## Transition from Markdown to JSON

**Phase 1**: Brief is presented in Markdown template format in chat/CLI. Humans review and approve in chat.

**Phase 2**: Brief is generated in both Markdown (for human readability) and JSON (for machine validation). Automated linters can validate JSON schema compliance and enforce struct constraints.

**Phase 2+ Automation**: JSON schema enables:
- Automated validation of brief completeness
- Machine-readable alternatives and trade-off analysis
- Automated constraint extraction and enforcement
- Cross-feature decision pattern detection
