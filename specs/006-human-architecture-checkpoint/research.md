# Phase 0 Research: Human Architecture Intent Checkpoint

**Feature**: 006-human-architecture-checkpoint  
**Branch**: `008-quality-profile-foundation`  
**Date**: 2026-05-09

## Overview

This document resolves technical unknowns and design decisions discovered during specification. Each research item identifies what was clarified, why it matters, and the chosen direction.

## Research Items

### R-1: Hook Invocation Mechanism and Blocking Semantics

**Unknown**: How is the checkpoint invoked within `/speckit.plan`? Is it a separate command hook, or does it integrate directly into the plan-generation script? How do we block task generation if human approval is not recorded?

**Research Summary**:
- **Decision**: The checkpoint integrates as a `before_plan` hook within the existing Speckit extension mechanism (`.specify/extensions.yml`).
- **Rationale**: The architecture intent checkpoint runs WITHIN the `/speckit.plan` command as an automatic pre-step, BEFORE task generation begins. This avoids creating a new separate command and keeps the workflow integrated.
- **Alternative Considered**:
  1. Create a new `/speckit.architecture-intent` command that runs before `/speckit.plan`. Rejected because this creates two separate invocations and requires users to understand a new workflow step.
- **Implementation Approach**: The `/speckit.plan` command automatically executes the checkpoint as its first phase:
  1. Loads the completed spec.md
  2. Invokes Squad to generate an architecture intent brief
  3. Surfaces the brief to the human for review and decision
  4. Records decisions in the plan context
  5. Validates that approval is recorded before proceeding to task generation
  6. If approval is not recorded, task generation does not proceed

### R-2: Architecture Intent Brief Structure and Generation

**Unknown**: What is the exact structure of the architecture intent brief? How is it generated by Squad? What format should it be in (markdown, JSON, structured conversation)?

**Research Summary**:
- **Decision**: The brief is a **structured Markdown document** generated by Squad and presented to the human in the chat/CLI context. The brief follows the structure defined in spec FR-001:
  - Intended architecture or design approach (1–2 paragraph summary)
  - Affected modules, services, files, or boundaries (list with brief justification)
  - Expected data model, API, CLI, UI, workflow, or storage changes (before/after or impact)
  - Important dependency, framework, or tooling choices
  - Security, reliability, migration, performance, and compatibility implications where relevant
  - Known assumptions and constraints
  - Meaningful alternatives considered (with brief trade-off analysis)
  - Explicit questions for decisions that require human preference or authority
  
  The brief is **not stored as a separate artifact** initially; it is presented in the chat/CLI conversation and then decisions are recorded in plan.md. Optionally, the brief template can be stored in `.specify/templates/architecture-brief-template.md` for future use.

- **Rationale**: Markdown is readable, version-controllable, and integrates naturally with Specrew's artifact chain. Presenting the brief in chat before storing decisions allows the human to ask clarifying questions and discuss trade-offs interactively. This respects the "minimal interruption" goal (US-3) by allowing flexibility in decision complexity.
- **Alternatives Considered**: Store the brief as a separate artifact (e.g., `architecture-brief.md`) in the spec directory. Rejected because it creates additional artifacts that must be kept in sync with plan.md decisions. Initial approach is lighter-weight: brief is chat-based, decisions are recorded in plan.md.
- **Implementation Approach**: Squad agent receives the spec.md as context and is prompted to generate a brief following the FR-001 structure. The agent presents the brief in chat. The human reads it, asks questions, and provides decisions. Decisions are then recorded by the human (or with Squad's help) in the Architecture Intent Review section of plan.md.

### R-3: Decision Recording Format and Ledger

**Unknown**: Where are decisions recorded? How do they persist? Can they be referenced during implementation? Is there a separate decision ledger, or do we use plan.md as the sole record?

**Research Summary**:
- **Decision**: **Phase 1 approach** (immediate): Decisions are recorded in the **Architecture Intent Review section of plan.md**. This is the authoritative record for this feature.
  - Accepted implementation direction (concise statement of the approved approach)
  - Rejected alternatives and reasons
  - Human constraints or forbidden paths (explicit list)
  - Unresolved architecture questions, if any
  - Links or references to decision ledger entries if the project uses `.squad/decisions.md`
  
  **Phase 2+ approach** (deferred): Optionally integrate with a dedicated decision ledger (e.g., `.squad/decisions.md`) for richer metadata, cross-feature traceability, and automated enforcement. This is out of scope for Phase 1.

- **Rationale**: Plan.md is the existing governance artifact for each feature; adding the Architecture Intent Review section keeps all decisions in one searchable place. This minimizes artifact sprawl and ensures decisions are reviewed with the plan. Future integration with a decision ledger is possible without breaking existing workflows.
- **Alternative Considered**: Create a separate `.squad/decisions.md` ledger at the repository level. Rejected for Phase 1 because it requires establishing a new artifact and governance process. Deferred to Phase 2.
- **Implementation Approach**: The Architecture Intent Review section is added to the plan-template.md. After the human approves the brief, the human and Squad collaboratively fill in this section with decisions, constraints, and alternatives. This becomes part of the permanent plan.md record.

### R-4: Constraint Enforcement and Conflict Detection

**Unknown**: How are recorded constraints enforced during implementation? If Squad later proposes a choice that conflicts with a recorded constraint (e.g., "do not use ORM" for a data model), how is this detected and escalated?

**Research Summary**:
- **Decision**: **Phase 1 approach**: Constraints are recorded in plan.md and visible to Squad agents during task generation and implementation. Enforcement is **manual and explicit**: Squad agents are instructed to check the Architecture Intent Review section of plan.md and incorporate constraints into task design and agent routing instructions. Drift is detected when constraints are visibly violated in generated tasks or implementation proposals.
  
  **Phase 2 approach** (deferred): Automated conflict detection can be added via a dedicated lens (decision-enforcement-lens) that checks task proposals against recorded constraints and flags violations.

- **Rationale**: Manual enforcement with visible constraint documentation is sufficient for Phase 1 and aligns with the "minimal automation" principle. Recording constraints explicitly makes them visible to all agents and humans reviewing the work. Automated checking is deferred to Phase 2 when metrics show it is needed.
- **Alternative Considered**: Enforce constraints via a rules engine or policy-as-code system. Rejected for Phase 1 because it requires building an evaluation framework. Deferred to Phase 2.
- **Implementation Approach**: Constraints are recorded clearly in plan.md. During task generation and implementation, Squad agents are instructed to read and incorporate these constraints into their decisions. If a conflict is discovered (e.g., a proposed implementation choice violates a recorded constraint), Squad must pause and return to the human with an explanation and alternatives. This escalation path is defined in plan.md as part of the drift/reconciliation gate (Constitution Check).

### R-5: Human Availability and Deadlock Handling

**Unknown**: What happens if the human is unavailable or does not respond to the checkpoint within a reasonable time? Does planning stall, or is there a timeout or escalation path?

**Research Summary**:
- **Decision**: **Phase 1 approach**:
  - The checkpoint waits for human input. If the human is unavailable, planning does not advance. This is intentional per the spec: "proceeding without approval is not allowed."
  - An **explicit escalation policy** must be defined and documented in plan.md for each project or team:
    - Option A: Wait until the human is available (suitable for teams with rotating on-call architects)
    - Option B: Escalate to a designated decision-maker or senior architect (suitable for larger teams)
    - Option C: Pause the feature and move to other work (suitable for agile teams with flexible prioritization)
  - The checkpoint provides a clear message to the human explaining what decisions are needed and why. If the human cannot decide, they can:
    1. Ask clarifying questions about the brief
    2. Request that Squad reconsider the proposed direction
    3. Delegate the decision to another team member (in which case that person becomes the decision-maker for the record)
    4. Record a "decision deferred" note with a target resolution date

- **Rationale**: Human approval is a required gate per the spec. Blocking planning when the human is unavailable is preferable to silently proceeding without approval. Explicit escalation policies ensure that teams have a clear path to resolution rather than deadlock.

- **Implementation Approach**: The checkpoint presents a clear, actionable brief to the human and waits for a decision. If the human does not respond, the checkpoint does not proceed. Teams define their escalation policy in governance documentation (e.g., team handbook, iteration retrospective). The checkpoint can include a timeout message that refers to the team's escalation policy, but the checkpoint itself does not auto-proceed.

### R-6: Integration with Pre-Implementation Approval Gate

**Unknown**: The spec states that the pre-implementation approval gate "remains required" (FR-006). How does the architecture intent checkpoint interact with the existing pre-implementation approval gate? Is the pre-implementation approval a separate review, or is it merged with the checkpoint approval?

**Research Summary**:
- **Decision**: The two gates are **separate and sequential**:
  1. **Architecture Intent Checkpoint** (new): Runs BEFORE task generation. Approves the high-level design approach. Human reviews the brief and approves or redirects architecture before planning continues.
  2. **Pre-Implementation Approval** (existing): Runs AFTER plan.md and tasks are generated. Approves the detailed plan and task breakdown before implementation begins. This gate must reference the accepted architecture intent so reviewers understand what architectural path was chosen.

  The pre-implementation approval summary (e.g., in a dedicated section of plan.md) MUST include:
  - Reference to the Architecture Intent Review section of plan.md
  - Confirmation that the plan and tasks align with the approved architecture intent
  - Any new risks or conflicts discovered during planning that require additional human decision

- **Rationale**: Separating the gates allows Squad to focus on implementation planning (tasks, dependencies, scheduling) without re-debating architecture. The pre-implementation approver can verify that the detailed plan honors the architecture decisions made at the checkpoint. This is more efficient than merging the two gates, which would require architecture experts to review all tasks.
- **Alternative Considered**: Merge the checkpoint and pre-implementation approval into a single gate. Rejected because it would require architecture experts to review task-level details, which is inefficient.
- **Implementation Approach**: After human approves the architecture intent checkpoint, planning continues and plan.md is generated. A new section is added to plan.md: **Pre-Implementation Approval Reference**, which documents:
  - The date and human who approved the architecture intent
  - Summary of the accepted direction and key constraints
  - Confirmation that the detailed plan aligns with the intent
  - Any new decisions discovered during planning that require approval
  The pre-implementation approval gate reviews this section and confirms alignment before authorizing implementation.

### R-7: Quality Profile and Tooling

**Unknown**: What quality profile or quality-assessment tooling applies to this feature? This feature modifies Spec Kit workflow, not application code, so standard stack-aware profiles may not apply.

**Research Summary**:
- **Decision**: A **custom quality composition** is used (not a standard preset). The composition focuses on:
  - **Workflow integration checks** (verify that the checkpoint hook integrates correctly with `/speckit.plan` and does not break existing workflows)
  - **Governance artifact consistency** (verify that Architecture Intent Review section is correctly formatted, complete, and traceable to decisions)
  - **Decision enforcement verification** (verify that recorded decisions are visible and enforceable during task generation and implementation)
  - **Test coverage for edge cases** (unavailable human, vague spec, conflicting directions)

- **Rationale**: This feature is a Spec Kit workflow modification, not an application library or service. Standard stack-aware profiles (e.g., "node public web service") do not apply. A custom composition allows us to focus on workflow integration, governance, and enforcement, which are the key quality concerns.
- **Implementation Approach**: Quality gates are defined in the plan.md Phase 1 Quality Planning section. Lenses are activated in Phase 2 for workflow integration, artifact consistency, and decision enforcement. Test cases are developed to cover user stories and edge cases (unavailable human, insufficient spec, discovery of conflicts during planning).
