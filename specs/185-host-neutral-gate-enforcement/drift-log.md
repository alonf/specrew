## D-001: PreToolUse gate empirically rejected - split-guard fired (2026-06-19)

**Decision point (plan boundary).** `plan.md` proposed activating the dormant gate seat via a Claude `PreToolUse` provider (FR-007), flagged as needing (1) empirical verification of the load-bearing product-domain assumption and (2) a maintainer nod, because it reverses F-184's deliberate decision to leave `PreToolUse` unregistered (~920 ms). The maintainer authorized activation ("solve it once and for all"); the empirical check ran first.

**Empirical measurement (per Write/Edit).** Cold pwsh spawn ~445 ms; lean gate (spawn + read state + parse) ~660 ms; loading `shared-governance.ps1` (179 KB) ~590 ms. The real per-Write cost is the **dispatcher spawn+load (fires on EVERY `PreToolUse`) + the gate-provider spawn = ~600 ms-1.2 s per Write/Edit.** This confirms the F-184 concern.

**The deeper finding.** `PreToolUse` hooks are **Claude-only**. But #2884's bug is **Antigravity** (a weak model) self-advancing. Antigravity has no `PreToolUse` surface, so the expensive Claude gate **does not touch the host that actually broke** - it would tax Claude (which respects the gates anyway; the F-184 finding is that strong models follow the gates and weak ones do not) to enforce where enforcement was not the problem.

**Conclusion.** The split-guard FIRES: the `PreToolUse` pre-block is both expensive and aimed at the wrong host. Do NOT silently proceed with FR-007 as planned. Surfaced to the maintainer for a design verdict (FR-007 approach change).

**Recommended pivot.** A cheap, cross-host **Stop-hook detection**: the Stop hook already runs every turn and captures verdicts (`HandoverStore.ps1`). Extend it to detect an unauthorized next-phase advance (a phase-N governance artifact written/committed without the phase-N verdict) and halt (the boundary stays un-authorized; the resume tells the next turn to revert and get the verdict). It works on Antigravity (which has a Stop hook), fires once per turn (no per-Write tax), and catches the actual bug. The Iteration-1 cleaning floor is committed regardless, so #2884 is already substantially better.

**RESOLVED (2026-06-19):** the maintainer chose the detection pivot (a), which then generalized into the unified design — see D-002. The Claude `PreToolUse` pre-block is dropped (it taxes the host that already behaves).

## D-005: every-stop packet enforcement moved cooperative-nudge -> STOP-BLOCK at the stop (maintainer ruling 2026-06-20)

This SUPERSEDES the D-003/D-004 "delivery ceiling: the stdout correction is a best-effort accelerator; the decision:block upgrade is deferred to the dogfood." The decision:block channel is no longer optional - it is the requirement.

**Research (verified, primary-source + adversarial-verify, `research/stop-block-capability-matrix.md`).** The maintainer asked "are you sure it is Claude-only?" - and the assumption was wrong. 5-host research (each host's official hook docs, an independent verifier opening every cited source): **4 of 5 hosts can BLOCK the turn-end** - Claude (`decision:block` + built-in `stop_hook_active`/8-cap), Codex (`decision:block` continuation, PR #14532, built-in `stop_hook_active`), Copilot (`agentStop decision:block`, fail-open, no built-in cap), Antigravity (`decision:continue`, soft, no built-in cap). **Only Cursor cannot** (observational `stop`; `followup_message` re-triggers a NEW turn - declared best-effort degrade). This is FR-004's capability matrix, now evidence-backed; the split-guard does NOT fire (bounded dispatcher extension, not a host-model rewrite).

**Implementation.** The conformance provider detects packet-absence and emits a host-agnostic block SENTINEL (`<<<SPECREW-STOP-BLOCK>>>` + the packet directive); the dispatcher translates it into the host's `StopBlockShape` envelope (new per-host `host.psd1` field = the FR-004 declaration) - `decision-block` / `decision-continue` / `followup-message`. Triggers: a BOUNDARY stop (HasPendingVerdict; carries the contiguous verdict marker) OR a SUBSTANTIAL (>=600-char) non-boundary hand-back post-intake; the design-workshop is excluded (pre-spec proxy). Loop-guard: the dispatcher honours `stop_hook_active` (allow when already continuing); the provider keeps its OWN consecutive-block cap (epoch-windowed counter, cap 3) for the hosts lacking a built-in one, degrading to a plain nudge over the cap so a session can NEVER hang. Strictly read-only to gate state; fully fail-open (any error/uncertainty -> allow the stop).

**Honest ceilings (carried).** (1) Cursor is best-effort (re-triggered turn; the human may glimpse the bare stop). (2) Capability != firing reliability - Codex Stop does not fire on Esc-interrupt / headless exec (real-host dogfood gate). (3) The design-analysis lens workshop AFTER spec.md is not excluded by the pre-spec proxy (dogfood-tunable). (4) The >=600-char substantial threshold is a crude proxy for "owes a packet" - dogfood-tunable; over-block cost is a forced packet (loop-capped), not a hang.

**Tests.** `conformance-detection.tests.ps1` (13 cases: boundary block + contiguous marker, substantial non-boundary block, pre-spec exclusion, false-positive guard, multi-gate marker, stale-packet-fires, relevant-packet-suppresses, loop-guard cap+reset, isolation, disabled) + `dispatcher-stop-block.tests.ps1` (per-host envelope x5, stop_hook_active allow, cursor degrade).
